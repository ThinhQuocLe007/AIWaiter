## 2. Payment System — Full Post-Refactor Audit

### 2.1 Schema Layer

```
PaymentRequest(table_id: str)                  → tool input
PaymentResponse(status, table_id, session_id,   → tool output
                qr_url, amount, message)
VerifyPaymentResponse(status, table_id, message) → tool output
PaymentStatus(PENDING | COMPLETED | FAILED)     → enum
```

All three Pydantic models are typed, exported from `schemas/__init__.py`, and used as `args_schema` in LangChain `@tool` decorators. The LLM never sees or needs `session_id` — it's resolved internally by the tool.

### 2.2 Service Layer

**`RestaurantDB`** (`services/restaurant_db.py`, 272 lines):
- 4 tables: `sessions` (ACTIVE/CLOSED), `orders` (CONFIRMED), `order_items` (per-row), `payments` (PENDING/COMPLETED)
- `get_active_session(table_id)` → latest ACTIVE session
- `get_orders_by_session(session_id)` → all orders with computed totals
- `add_payment(session_id, amount, qr_url)` → insert PENDING payment
- `update_payment_status(session_id, "COMPLETED")` → cascading session close
- No `DROP TABLE` — uses `CREATE TABLE IF NOT EXISTS`

**`PaymentManager`** (`services/payment_mgr.py`, 42 lines):
- `generate_qr_payload(session_id, amount)` → VietQR image URL
- `schedule_mock_verify(session_id)` → `threading.Timer(5s)` auto-completes payment
- `_auto_complete()` → creates fresh `RestaurantDB`, marks COMPLETED, closes session

### 2.3 Tool Layer

| Tool | Input Schema | Key Behavior | Status |
|------|-------------|-------------|--------|
| `request_payment` | `PaymentRequest(table_id)` | Resolves session, sums orders, generates QR, schedules 5s mock verify | ✅ |
| `verify_payment` | `table_id: str` | Resolves session, checks payment, marks COMPLETED if pending | ✅ |
| `confirm_order` | `table_id, items[]` | Creates session, looks up prices, persists order+items | ✅ |

### 2.4 Remaining Payment Gaps

| # | Issue | Risk | Fix |
|---|-------|------|-----|
| 1 | **Mock auto-verify assumes payment always succeeds** — no failure path, no retry | Medium | Add random failure mode (20% chance) for realistic testing |
| 2 | **No idempotency on `request_payment`** — calling twice generates two different QR URLs for the same session | Low | Cache QR by session_id or hash cart contents |
| 3 | **`threading.Timer` for mock verify leaks if process exits mid-timer** | Low | Use `atexit` cleanup or switch to asyncio |
| 4 | **Bank credentials hardcoded** (`ICB`, `123456789`) | Low | Extract to `payment_config.py` / `.env` |
| 5 | **No webhook endpoint** — real VietQR IPN callbacks can't land anywhere | High (prod) | Add FastAPI webhook route (Phase 5+) |
| 6 | **Payment worker has no error handling** — Ollama crash = graph crash | Medium | Add try/except around LLM invoke |

### 3.2 Voice State Machine

```
idle ──[openPanel()]──→ listening ──[2.2s]──→ thinking ──[3.9s]──→ speaking
  ↑                                                                      │
  └──────────────────[closePanel() / stop()]─────────────────────────────┘
                              │
                    confirmRecommendation() → cart.addItem()
```

Currently driven by `MOCK_TURNS[]` array (3 scripted conversations). `startListening()` cycles through mock turns with `setTimeout` timers.

### 3.3 Integration Surface with the Brain

The frontend currently has **zero connection** to the Python brain. It:
- Loads menu data **statically** from `menu.json` via `menuAdapter.ts` (no API fetch)
- Has no `fetch()` or `WebSocket` calls to any backend
- Uses mock timers for voice (no Web Speech API, no LLM endpoint)
- Cart lives only in Pinia memory (lost on page refresh)

**To integrate**, the frontend needs:
1. A backend URL (FastAPI on Jetson or server) serving `/api/menu`, `/api/state/{table_id}`
2. WebSocket endpoint for real-time cart/order/payment state updates
3. Replace `MOCK_TURNS` with actual Web Speech API → text → WebSocket → LLM → WebSocket → TTS
4. Replace static `menuAdapter.ts` with `fetch('/api/menu')` call

### 4.2 State Updater — `isinstance` Checks

```python
# graph.py:state_updater_node
if isinstance(content, SyncCartResponse) and content.status == "success":
    state["active_cart"] = content.cart
    state["order_stage"] = "AWAITING_CONFIRMATION"
elif isinstance(content, ConfirmOrderResponse) and content.status == "success":
    state["order_stage"] = "CONFIRMED"
elif isinstance(content, PaymentResponse) and content.status == "success":
    state["payment_qr_url"] = content.qr_url
    state["payment_amount"] = content.amount
```

This replaces the old fragile string-matching (`"SYNC_CART_SUCCESS:" in content`). The validator now also checks order_stage transitions:
- `confirm_order` rejected unless `order_stage == "AWAITING_CONFIRMATION"`
- `request_payment` requires `table_id` argument present

**Pattern**: 2 failures are SLM singleton misclassification (CHAT instead of SEARCH). 6 failures are multi-intent dispatch bugs.

**Critical insight**: The SLM correctly classifies multiple intents in all 6 multi-intent cases. The failures happen because the `route_to_worker()` greedy pick only dispatches the highest-priority intent. But wait —

### 5.3 The Multi-Intent Dispatch Fix (Already Implemented)

The `_route_after_updater()` function in `graph.py` now shifts `current_intents[1:]` and routes to the next worker. This means multi-intent dispatch SHOULD work. But the eval results from June 2 still show 6 multi-intent failures.

**Possible explanations**:
1. The eval was run BEFORE the June 4-5 refactor — the fix wasn't deployed yet
2. The eval maps `expected` vs `predicted` at the routing level (single intent), not the execution level
3. The `eval_router.py` script checks only the FIRST dispatched intent, not whether all intents get processed

**To verify**: Re-run `eval_router.py` on the current codebase. If multi-intent dispatch now works, accuracy should jump from 90% to ~97.5%.
