## 0. Distributed System Architecture

### 0.1 Physical Deployment

```
┌─────────────────────────────┐              MQTT              ┌──────────────────────────────────┐
│    SERVER (AI Brain)        │◄─────────────────────────────►│  JETSON ORIN 8GB (Mobile Robot)  │
│                             │                               │                                  │
│  AIWaiterGraph              │  ai_waiter/{table}/speech →   │  Silero VAD → PhoWhisper STT    │
│  (LangGraph state machine)  │  ← ai_waiter/{table}/response │  TTS Engine (edge-tts / XTTS-v2) │
│  Ollama (qwen2.5, llama3.1) │  ← ai_waiter/{table}/state    │  SLAM + Nav2 (auto-navigate)     │
│  SQLite (orders, checkpoints)│ ← ai_waiter/{table}/action    │  FastAPI (serve web UI to monitor)│
│  MQTT Client (paho-mqtt)    │                               │  MQTT Client (paho-mqtt)         │
│                             │                               │  Monitor (customer-facing web UI) │
└─────────────────────────────┘                               └──────────────────────────────────┘
```

### 0.2 Why This Split

| Component                     | Runs On    | Rationale                                                                                                             |
| ----------------------------- | ---------- | --------------------------------------------------------------------------------------------------------------------- |
| AIWaiterGraph + Ollama LLMs   | **Server** | Needs GPU/RAM for 3B+7B models. Jetson 8GB can't run SLM + worker LLMs simultaneously with SLAM                       |
| Silero VAD + PhoWhisper STT   | **Jetson** | Audio capture is local to the robot. Sending raw audio over network is wasteful. Transcribe on-device, send text only |
| TTS Engine                    | **Jetson** | Audio playback is local. Synthesize on-device from text received via MQTT                                             |
| SLAM + Nav2                   | **Jetson** | Requires local sensor data (LiDAR, camera). Standard ROS 2 navigation stack                                           |
| Web UI (FastAPI + HTML)       | **Jetson** | The monitor is physically attached to the robot. Serve locally, reachable at `localhost:8000` on Jetson               |
| MQTT Broker (e.g., Mosquitto) | **Server** | Single broker for all robot-server communication. Lightweight, battle-tested for IoT/robotics                         |

### 0.3 MQTT Topic Design

| Topic                        | Direction       | QoS | Payload                                                                              |
| ---------------------------- | --------------- | --- | ------------------------------------------------------------------------------------ |
| `waiter/{table_id}/speech`   | Jetson → Server | 1   | `{"text": "Cho 2 Phở Bò", "timestamp": 1717400000.0}`                                |
| `waiter/{table_id}/response` | Server → Jetson | 1   | `{"text": "Dạ...", "state_diff": {...}, "is_final": true}`                           |
| `waiter/{table_id}/state`    | Server → Jetson | 0   | `{"active_cart": {...}, "order_stage": "DRAFTING", "payment_qr_url": "https://..."}` |
| `waiter/{table_id}/action`   | Jetson → Server | 1   | `{"type": "confirm_payment", "table_id": "T1"}` — user actions from web UI           |
| `waiter/{table_id}/status`   | Jetson → Server | 0   | `{"battery": 85, "location": "table_5", "nav_status": "arrived", "is_active": true}` |
| `waiter/{table_id}/tts_done` | Jetson → Server | 0   | `{"played": true}` — TTS playback confirmation                                       |

**QoS rationale**: QoS 1 for commands/speech (at-least-once delivery), QoS 0 for state updates (high-frequency, loss-tolerable — next message replaces stale one).

### 0.4 End-to-End Conversation Flow

```
1. Customer speaks → Jetson mic → Silero VAD detects speech
2. PhoWhisper transcribes → Vietnamese text
3. MQTT → "waiter/T1/speech" → Server receives text
4. Server: AIWaiterGraph.chat(text, table_id="T1")
5. Server: LangGraph processes (router → worker → tools → state_updater → loop)
6. Server → MQTT "waiter/T1/response" → {"text": response, "is_final": true}
7. Server → MQTT "waiter/T1/state" → {"active_cart": {...}, "order_stage": ...}
8. Jetson receives response → TTS synthesizes → speaker plays audio
9. Jetson FastAPI receives state → pushes to browser via WebSocket
10. Browser renders: updated cart, QR code, order stage indicator
```

### 2.3 Routing Priority Bug — Multi-Intent Dispatch

`route_to_worker()` (line 70-84) uses hardcoded priority:

```python
# Priority: ORDER > PAYMENT > SEARCH > CHAT
if any(i in ("ORDER", "ORDER_CONFIRM") for i in intents):
    return "order_worker"
elif "PAYMENT" in intents:
    return "payment_worker"
elif "SEARCH" in intents:
    return "search_worker"
return "chat_worker"
```

**Problem**: If SLM returns `["ORDER", "PAYMENT"]`, only `order_worker` is dispatched. PAYMENT is silently dropped. The user has to ask "tính tiền" again after ordering completes. This causes 6 of 8 eval failures.

**Fix**: Need `pending_intents` queue in `AgentState` with sequential dispatch (Sub-06 from problem_06 plan).

### 2.4 State Updater Gap — No Payment Handling

`state_updater_node` (line 20-42) only handles two tool results:

```python
if "SYNC_CART_SUCCESS:" in content:     # → updates active_cart, order_stage
elif "CONFIRM_ORDER_SUCCESS" in content: # → updates order_stage
# Nothing for payment tools!
```

When `request_payment` returns a QR URL, the `state_updater` sees the tool name `"request_payment"` in `route_after_updater` and correctly routes back to `payment_worker`, but no state fields are updated. The QR URL and payment status remain invisible to any external observer.

## 4. Input/UX — Perception, TTS & Streaming (Jetson-Side)

### 4.1 Current State

User interaction path: **Python strings in `main.py` → `print()` to stdout**. That's it.

```
Current:  main.py (hardcoded text) → agent.chat() → print(response) → stdout
Target:   Microphone → VAD → STT → MQTT → Server Brain → MQTT → TTS → Speaker
          Browser → FastAPI → MQTT → Server Brain → MQTT → FastAPI → WebSocket → Browser
```

### 4.2 Why Perception & TTS Run on Jetson (Not Server)

| Decision             | Rationale                                                                                                                                                                                                               |
| -------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| STT on Jetson        | Sending raw audio over MQTT is wasteful (16kHz × 16bit = 256kbps continuous). Transcribe on-device, send ~100 bytes of text per utterance. PhoWhisper (tiny/small) runs in ~500MB RAM on Jetson.                        |
| TTS on Jetson        | Audio playback is physically local to the robot's speaker. Synthesize from text received via MQTT. No need to stream audio over network. edge-tts requires internet; XTTS-v2 runs fully local on Jetson if RAM permits. |
| Web server on Jetson | The monitor is physically attached. FastAPI serves `localhost:8000` on Jetson — no network latency, no CORS. Browser connects via WebSocket for real-time state sync.                                                   |
| Brain on Server      | Ollama loads qwen2.5:3b (~2GB) + llama3.1 (~4GB) simultaneously. Jetson 8GB can't run both LLMs + SLAM + Nav2 + STT + TTS. Server-side brain also enables multi-robot scale later.                                      |

### 4.3 Perception Pipeline (Jetson)

| Component                 | File                             | What It Does                                                                                                                         | Dependencies                    |
| ------------------------- | -------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------- |
| Voice Activity Detection  | `perception/vad_silero.py`       | Detects speech boundaries in audio stream, outputs clean speech chunks. Silero VAD runs at <5ms latency on Jetson CPU                | `torch`, `silero-vad`           |
| Speech-to-Text            | `perception/stt_phowhisper.py`   | Transcribes Vietnamese audio chunks to text. PhoWhisper (fine-tuned Whisper for Vietnamese) — use `tiny` or `small` model for Jetson | `faster-whisper`, `sounddevice` |

### 4.5 State-Aware Streaming (Sentence-Level)

The LLM response comes as full text from the server via MQTT. Jetson splits by sentence and synthesizes sequentially:

```python
# Jetson-side: MQTT callback for waiter/{table}/response
async def on_response_received(topic: str, payload: dict):
    text = payload["text"]
    state_diff = payload.get("state_diff", {})
    state = payload.get("state", {})

    # 1. Push state update to browser via WebSocket
    if state:
        await ws_manager.broadcast(table_id, {"type": "state_update", "data": state})

    # 2. Adapt TTS tone based on agent state
    stage = state.get("order_stage", "IDLE")
    tts_config = {
        "IDLE":                {"speed": 1.0, "pitch": 0},
        "DRAFTING":            {"speed": 1.1, "pitch": 0},
        "AWAITING_CONFIRMATION": {"speed": 0.95, "pitch": 0},
        "CONFIRMED":           {"speed": 1.0, "pitch": +2},
    }.get(stage, {"speed": 1.0, "pitch": 0})

    # 3. Sentence-level streaming: play first sentence while rest synthesize
    sentences = split_vietnamese_sentences(text)
    for i, sentence in enumerate(sentences):
        if speech_interrupted:  # Customer started speaking again
            break
        audio = tts.synthesize(sentence, **tts_config)
        audio_player.play(audio)  # Non-blocking for subsequent sentences
```

**Two streaming levels:**

| Level              | How It Works                                                                                                               | Latency Benefit                                             | Difficulty                                  |
| ------------------ | -------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------- | ------------------------------------------- |
| **Sentence-level** | Server sends full response text via MQTT. Jetson splits by sentence → TTS plays first sentence while synthesizing the rest | User hears first sentence ~500ms after server response      | Easy — implement first                      |
| **Token-level**    | Server streams tokens via MQTT as LLM generates. Jetson buffers until sentence boundary (`.`, `?`, `!`, `ạ`) → starts TTS  | User hears start of response before LLM finishes generating | Hard — requires LangGraph streaming support |

**Recommendation**: Implement sentence-level first. Token-level is nice-to-have; the latency difference on Jetson is marginal compared to TTS synthesis time (~200-400ms per sentence).

### 4.6 Web Server on Jetson (FastAPI)

The web server runs on Jetson, not server. It serves the HTML frontend and bridges browser ↔ MQTT:

**Endpoints (Jetson FastAPI):**

| Method      | Path                    | Purpose                                                                                   |
| ----------- | ----------------------- | ----------------------------------------------------------------------------------------- |

```
└── L10:   Graceful fallback when tools/prompts fail (server)
```

### Quick-Start Order (What to Build First)

If you want to get the full pipeline working end-to-end quickly:

1. **Phase 1** (1-2 days) — fix setup.py, OrderDB, ROOT path
2. **Phase 4** (1 day) — MQTT bridge (just pass text through, no processing)
3. **Phase 5** (2 days) — FastAPI + frontend on Jetson (static cart display)
4. At this point you can type text in a browser on the monitor, it goes MQTT → server → MQTT → browser, and you see responses
5. **Phase 2** (2 days) — multi-intent dispatch, payment state fields (fix eval failures)
6. **Phase 6** (3-4 days) — VAD, STT, TTS on Jetson
7. At this point you have full voice → brain → voice + web UI
