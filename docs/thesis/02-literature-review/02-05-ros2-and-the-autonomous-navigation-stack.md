# 2.5 ROS2 and the Autonomous Navigation Stack

The robot's low-level motion, described in the previous sections, is only the foundation on which autonomous behavior is built. To perceive its surroundings, build and reuse a map, plan a path, and drive safely to a goal, the robot needs a larger software system that ties all of its sensors and algorithms together. This section describes that system at the level of general theory, beginning with the middleware that connects the software components, then the way the robot builds and later reuses a map of its environment, the navigation stack that turns a goal into safe motion, and finally the simulator used to develop and test the software before it runs on the physical robot. The focus throughout is on the general principles behind each part of the stack, not on the specific settings, maps, and results obtained on the actual robot.

### 2.5.1 Robot Operating System (ROS2)

The software of a modern robot is not a single program but many small programs that must run at the same time and constantly exchange information - one reading the LiDAR, another computing odometry, another planning a path, another driving the motors. Writing all of this as one monolithic application would be fragile and hard to reuse, so instead it is built on a robotics middleware. ROS2 (Robot Operating System 2) is such a middleware: despite its name it is not an operating system in the usual sense, but a framework that provides the communication, tools, and conventions that let many independent programs cooperate as one robotic system.

In ROS2, each independent program is called a node, and a running robot is a collection of nodes. Nodes do not call each other directly; instead they communicate through a small number of well-defined patterns. The most common is the topic, a named channel onto which one node publishes messages and from which any number of other nodes subscribe - for example, the LiDAR driver publishes scans on a topic while the mapping and navigation nodes subscribe to them. This publish–subscribe model keeps the nodes loosely coupled: a node only needs to know the name and type of a topic, not which other nodes are producing or consuming it. For request-and-reply interactions ROS2 provides services, and for long-running goals that report progress and can be cancelled - such as "navigate to this point" - it provides actions. Underneath these patterns, ROS2 uses an industry-standard communication layer called DDS (Data Distribution Service), which handles discovery and message delivery across the network and removes the single central master that earlier versions of ROS depended on.

Because a robot carries many sensors and moving parts, each with its own position and orientation, ROS2 also standardizes how spatial relationships are described. The TF (transform) system keeps track of every coordinate frame on the robot - the world, the map, the robot body, and each sensor - and of how they are related over time, so that a measurement taken in one frame, such as a point seen by the camera, can be expressed in any other frame, such as the map. The physical layout of the robot itself - its links, joints, wheel spacing, and sensor mounting positions - is described in a URDF (Unified Robot Description Format) file, which supplies both the TF tree and the geometric model used for visualization and simulation.

ROS2 was chosen over the original ROS for this work because of these properties: the DDS-based communication removes the central point of failure, provides more reliable and configurable message delivery, and is better suited to real-time and multi-machine operation, while the large ecosystem of ready-made packages - sensor drivers, SLAM, and navigation - greatly reduces the amount of code that must be written from scratch.

### 2.5.2 SLAM and RTAB-Map

Before a robot can navigate a room on its own, it needs a map of that room and it needs to know where it is on that map — and the two needs are entangled: building an accurate map requires knowing where each measurement was taken from, while knowing where the robot is requires a map to compare against. Solving both at once is known as SLAM (Simultaneous Localization and Mapping). A SLAM system is usually organised into two cooperating parts: a front end that processes the raw sensor data (LiDAR scans, camera frames, odometry) into motion estimates and spatial constraints, and a back end that keeps the whole trajectory and map globally consistent by absorbing the noise and small errors the front end inevitably introduces.

Depending on the sensors used by the front end and the mathematics used by the back end, several families of SLAM exist; the main ones relevant to an indoor mobile robot are summarised below. Regarding the back end, older filter-based methods (EKF-SLAM, particle filters) update a single running estimate recursively, while modern graph-based methods store the pose history as a graph and optimise it as a whole; because they can revisit and correct earlier poses, graph-based methods produce more globally consistent maps and are now the dominant approach.

For this robot, no single sensor is ideal on its own: the 2D LiDAR gives accurate geometry in any lighting but sees only one horizontal plane and can make two similar-looking places hard to tell apart, while the RGB-D camera provides the visual appearance needed to recognise a previously visited place. This work therefore uses a hybrid, graph-based system built for exactly this combination — RTAB-Map (Real-Time Appearance-Based Mapping) [n] — with a clear division of labour: the LiDAR supplies the geometry of the map, and the camera makes that map globally consistent through place recognition. The map the robot builds and navigates on remains two-dimensional.

RTAB-Map represents the map as a graph whose nodes are the poses at which the robot observed the environment (with the LiDAR and camera data captured there) and whose edges are spatial constraints between those poses. It does not estimate the robot's motion itself: it takes an external odometry stream — here, the fused encoder-and-IMU estimate of the previous section — as the backbone of the graph, and refines the resulting links by scan matching with the ICP (Iterative Closest Point) algorithm, which iteratively shifts and rotates one LiDAR scan against a neighbouring one until the two fit, yielding a more precise relative transform than odometry alone. Since the robot moves on a flat floor, the whole estimation is constrained to the plane (three degrees of freedom: x, y, θ).

The feature that makes the graph approach powerful is loop closure. As drift accumulates, the estimated trajectory slowly bends away from the truth, so when the robot returns to a place it has visited, its estimated position no longer matches. RTAB-Map detects such revisits from appearance: each camera image is described by its distinctive visual features, quantised into a vocabulary of "visual words" (the bag-of-words technique), and compared against previously visited places; a hypothesised match is then verified and refined geometrically with the same LiDAR ICP before being added to the graph as a new edge. The camera finds the loop, the LiDAR measures it precisely. The new constraint exposes the accumulated drift, which the back end removes by re-optimising all poses at once, pulling the trajectory back into global consistency; proximity detection adds further constraints by scan-matching directly against nearby mapped nodes even when the camera view has changed. To stay real-time as the graph grows, RTAB-Map also employs a memory-management scheme that keeps only recent and frequently seen nodes in a working memory searched every cycle, moving the rest to long-term storage. The end product used by the rest of the stack is a single 2D occupancy grid — a top-down map of free and blocked space that the navigation stack plans on — built from the LiDAR, with the camera contributing only indirectly through the loop closures that keep it consistent.

### 2.5.3 Localization on a Known Map

Once a map of the environment has been built and saved, the robot no longer needs to map and localize at the same time; on later runs it only needs to locate itself within the already-known map. In this localization mode the robot compares its live sensor readings against the stored map to estimate its current pose, and the map itself is kept fixed rather than extended.

This work performs localization with the same RTAB-Map framework used for mapping, run in its localization mode: the saved graph is loaded and no new nodes are permanently added, so the robot works against a frozen map. Localizing then reuses the very mechanisms that built the map. When the robot starts up or becomes uncertain, the appearance-based place recognition described earlier - the bag-of-words matching of the current camera view against the stored ones - tells it which part of the map it is in, in the same way a loop closure is detected during mapping. Once roughly placed, the estimate is refined geometrically: the current LiDAR scan is matched against the scans stored in the nearby graph nodes (the same ICP scan matching used to refine edges during mapping), pinning the pose down precisely. Using one framework for both mapping and localization keeps the two consistent - the robot localizes against exactly the data structure it built.

A widely used alternative for localization on a fixed map is AMCL (Adaptive Monte Carlo Localization), a particle-filter method that maintains many hypotheses about where the robot might be - a cloud of candidate poses - and, as new LiDAR readings arrive, keeps the hypotheses that match the map well and discards those that do not, so that the cloud gradually converges on the true pose. AMCL, however, typically needs to be told roughly where the robot starts, whereas the appearance-based approach can recognize the starting place from what the camera sees; this is one of the reasons the RTAB-Map localization mode is used in this work.

### 2.5.4 Navigation (Nav2)

Knowing where it is and having a map is still not enough for a robot to move on its own: it must also decide how to get from its current pose to a goal, and it must do so while avoiding obstacles, including ones that were not present when the map was built. In ROS2 this is the job of Nav2, the standard navigation stack, which coordinates a set of specialized servers - chiefly a planner, a controller, and a behavior manager - to turn a single goal pose into a continuous stream of safe velocity commands.

Nav2's reasoning about free and blocked space is organized around costmaps, which are grids that mark each cell with how costly or dangerous it is to occupy. A global costmap covers the whole known map and is used for long-range planning, while a local costmap covers a small window that moves with the robot and is continually refreshed from the live LiDAR scans, so that newly appearing obstacles - a person, a chair pulled into the aisle - are represented immediately. Around every obstacle the costmap adds an inflation layer, a margin of raised cost that fades with distance, so that planned paths naturally keep a safe clearance rather than grazing the walls and furniture.

On top of these costmaps, planning happens at two levels. A global planner searches the global costmap for a complete, roughly optimal route from the robot to the goal, using a classic graph-search algorithm such as Dijkstra's or A*. A local controller then turns that route into actual velocity commands. This work uses a pure-pursuit style controller, whose idea is simple and well suited to a differential-drive robot: the controller picks a lookahead point - a point on the planned path a short distance ahead of the robot - and continuously steers toward it, computing the turning rate that would carry the robot along the arc connecting its current pose to that point. As the robot advances, the lookahead point slides forward along the path, so the robot smoothly "chases" the path rather than tracking it point by point. The regulated variant used here additionally slows the robot down in tight turns and near obstacles detected in the local costmap, and rotates in place toward the path heading when needed. The controller respects the robot's differential-drive nature throughout - it never commands sideways motion ( = 0) and it can turn in place, matching the non-holonomic constraint derived earlier. Finally, a behavior layer supervises the whole process and invokes recovery behaviors, such as backing up, rotating to clear the sensors, or re-planning, whenever the robot becomes stuck, so that a single blocked path does not end the task.

### 2.5.5 Robot Simulation with Gazebo

Developing and testing navigation software directly on a physical robot is slow and risky: a mistake can drive the robot into a wall, a mishandled sensor takes real time to reproduce, and every experiment requires physically setting up the environment. For this reason the software is first developed and exercised in simulation, and only then transferred to the real robot. This work uses Gazebo, a physics-based robot simulator that integrates closely with ROS2.

Gazebo reproduces the robot and its environment inside a virtual world with a physics engine, so that gravity, contact, friction, and the robot's own dynamics behave realistically. Crucially, it also provides sensor plugins that generate synthetic data in exactly the same message formats the real sensors would produce: a LiDAR plugin emits laser scans, a depth-camera plugin emits RGB and depth images, an IMU plugin emits angular velocity and acceleration, and a differential-drive plugin accepts velocity commands and moves the simulated robot accordingly while reporting wheel odometry. Because these simulated streams look the same to the rest of the software as real ones, the same ROS2 stack - the encoder–IMU odometry fusion, RTAB-Map SLAM and localization, and Nav2 navigation - runs unchanged against either the simulator or the physical robot. This lets the whole system be developed, debugged, and tuned safely in simulation, with no risk to the hardware and with full control over the test environment, and then deployed on the real platform running the same configuration.
