A robot that delivers food in a restaurant must answer one question repeatedly: *where should I go next?* The answer changes with every business event — a party is seated at table 3, an order for table 5 is marked complete in the kitchen, the session at table 2 is paid and the table is vacated. The navigation target is not fixed; it is a function of live restaurant state managed by an external AI agent. This section surveys the technologies that make autonomous navigation possible — odometry, sensor fusion, SLAM, path planning, and precision docking — and examines prior academic ROS2 delivery robot systems. The recurring finding is that each technology is individually mature, but all prior systems assume navigation goals are pre-set. The specific gap is coupling these mature navigation capabilities to a goal source that is external, AI-driven, and dynamic.

### 2.2.1 Wheel Odometry and Sensor Fusion

A differential-drive robot estimates its motion through two complementary sensor modalities: wheel encoders provide relative displacement, and an inertial measurement unit (IMU) provides angular rate and linear acceleration. Neither alone is sufficient. Encoders accumulate drift that grows unbounded with distance — each wheel slip on a smooth floor, each uneven tile, each in-place rotation introduces error that no subsequent encoder reading can correct [2.2.1]. An IMU provides absolute angular rate independent of wheel-ground contact, but consumer-grade MEMS IMUs exhibit gyro bias — a constant offset that integrates into growing angular error over time — and accelerometer noise that makes double-integration to position unreliable for anything beyond a few seconds [2.2.2].

**Encoder odometry.** For a differential-drive platform with wheel encoders producing P pulses per revolution and a gear ratio G, the distance traveled per encoder tick is determined by the wheel diameter D and the total ticks per revolution N = P × 4 × G. Wheel velocities are computed from tick counts over a fixed time interval, and forward kinematics gives the robot's body-frame velocities [2.2.3]:

$$V_x = \frac{V_A + V_B}{2}, \quad V_\omega = \frac{V_B - V_A}{W}$$

where $V_A$ and $V_B$ are the left and right wheel linear velocities and $W$ is the wheel track width. Integration over time yields pose. The fundamental limitation is that encoder odometry measures wheel rotation, not wheel displacement — slip, skid, and uneven terrain produce motion that the encoders do not capture. On a smooth restaurant floor with hard wheels, slip during acceleration and in-place rotation is the dominant error source.

**Inertial measurement.** The IMU — in this class of platform, typically an MPU6050 or equivalent 6-DoF MEMS sensor — outputs angular velocity (gyroscope, deg/s) and linear acceleration (accelerometer, m/s²) as raw 16-bit integer values [2.2.4]. Conversion to SI units requires applying factory-calibrated scale factors. The gyroscope's angular rate can be integrated once to produce relative heading change — distinct from the encoder-derived heading, which degrades during wheel slip. The accelerometer measures the gravity vector when the robot is stationary, providing an absolute tilt reference, and can detect linear acceleration during motion, though double-integration to position is too noisy to be useful beyond very short time windows.

**Sensor fusion via the Extended Kalman Filter.** The EKF addresses the complementary nature of these sensors: encoders are accurate over short straight-line segments but degrade during rotation and slip; the IMU gyroscope provides clean angular rate regardless of wheel-ground contact but drifts over time due to bias [2.2.5]. An EKF maintains a state vector — typically $[x, y, \psi, \dot{x}, \dot{y}, \dot{\psi}]$ for a planar robot — and alternates between prediction (advance the state using a motion model driven by one sensor) and update (correct the state using measurements from another sensor). The `robot_localization` package provides a configurable EKF implementation for ROS2, supporting `two_d_mode` which constrains the state to planar motion [2.2.6]. The typical fusion configuration for a differential-drive robot feeds encoder velocities to the prediction step and IMU angular velocity to the update step. IMU yaw is not fused in the absence of a magnetometer, because integrated gyro yaw drifts independently of the EKF state.

**Prior work on EKF for differential-drive platforms.** The EKF has been the standard approach for wheel-IMU fusion in mobile robotics since the foundational work of Thrun, Burgard, and Fox [2.2.7]. Moore and Stouch demonstrated `robot_localization` on low-cost platforms with consumer-grade sensors, establishing that the EKF can produce usable odometry from inexpensive hardware when covariance parameters are properly tuned [2.2.8]. University projects have validated EKF-fused odometry on custom differential-drive platforms in controlled lab environments — typically a short rectangular path with measured start-end error.

**→ Gap.** Prior work validates EKF odometry in a single context: the robot travels a closed path and the start-end error is measured. Restaurant service introduces a different error regime. The robot executes multiple kitchen→table→kitchen cycles per session. Each cycle involves acceleration from a dead stop, a straight segment of 3–5 meters, a deceleration to near-zero at the table approach zone, an in-place rotation to face the table, and then the same sequence in reverse. The cumulative drift across multiple cycles — not the per-cycle error — determines whether the robot can re-detect the ArUco marker at the docking zone on cycle N. If the cumulative yaw error exceeds the D435 camera's horizontal field of view at the expected detection distance (approximately 1 meter), the ArUco re-localization fails and the robot cannot dock. No prior work has evaluated EKF-fused odometry under repeated restaurant service cycles with cumulative drift tracking across cycles, using the specific sensor combination of consumer-grade MPU6050-class IMU and encoder-equipped DC motors on a purchased TWD platform.

### 2.2.2 SLAM and Map Building

Simultaneous Localization and Mapping (SLAM) addresses the chicken-and-egg problem of autonomous navigation: to localize, the robot needs a map; to build a map, the robot needs to know where it is [2.2.9]. Modern SLAM systems separate the problem into a front-end — sensor data processing and frame-to-frame alignment — and a back-end — pose graph optimization with loop closure detection — that jointly estimate the robot's trajectory and the environment map.

**LiDAR-based SLAM.** A 2D LiDAR such as the RPLiDAR A2M8 produces a planar scan of range measurements at a fixed angular resolution — typically 360 points per revolution at 5–10 Hz [2.2.10]. Sequential scans are aligned via scan matching algorithms, most commonly Iterative Closest Point (ICP), which finds the rigid transform that minimizes the distance between corresponding points in consecutive scans. The aligned scans accumulate into an occupancy grid — a 2D array where each cell stores the probability of being occupied. LiDAR SLAM is robust to lighting conditions and produces geometrically accurate maps of the LiDAR plane, but it can fail in environments with few distinct geometric features (long straight corridors) where scan matching produces ambiguous alignments.

**Visual SLAM.** An RGB-D camera such as the Intel RealSense D435 provides both color images and registered depth maps at 30 Hz [2.2.11]. Visual SLAM extracts feature points — corners, blobs, edges — from consecutive images, matches them across frames, and estimates camera motion from the feature correspondences. The key advantage over pure LiDAR is loop closure: when the robot returns to a previously visited location, visual feature matching detects the revisit even if the LiDAR scan is geometrically ambiguous. Bag-of-words models — where visual features are quantized into a vocabulary of visual words — enable efficient large-scale place recognition independent of the robot's current pose estimate [2.2.12].

**RTAB-Map (Real-Time Appearance-Based Mapping).** RTAB-Map is a graph-based SLAM framework that fuses LiDAR geometry with RGB-D visual features [2.2.13]. It builds a pose graph where nodes represent robot poses with associated sensor data (LiDAR scan, RGB image, depth map) and edges represent spatial constraints — odometry between consecutive poses, loop closures between revisited locations. The framework employs a memory management mechanism inspired by human short-term and long-term memory: recent nodes and features are kept in working memory for real-time processing; older data is transferred to long-term memory and retrieved when a loop closure candidate is detected. Graph optimization — typically using g²o or GTSAM — adjusts all node poses to minimize constraint errors, producing a globally consistent map. RTAB-Map outputs a 2D occupancy grid map suitable for Nav2 navigation alongside a feature database for localization.

**Prior work on restaurant environment mapping.** University projects using ROS2 delivery robots have mapped campus cafeterias, hospital wards, and office corridors using a combination of Cartographer or RTAB-Map [2.2.14]. These maps serve navigation: the robot knows the occupancy grid and can plan collision-free paths. What they do not provide is semantic structure — the map contains a hallway with alcoves, but it does not know which alcove is "table 3." The waypoint-to-table mapping is performed manually by a human operator who records the pose of each table during or after mapping and stores it in a configuration file.

**→ Gap.** Prior work builds maps for navigation and treats table waypoints as static configuration. Two related problems remain unsolved for a restaurant AI system. First, the table waypoint configuration must be queryable by an external backend — the dispatcher must be able to resolve `table_id=3` to a `(x, y, yaw)` goal pose at runtime without human intervention. RTAB-Map does not expose a table-semantic query layer; it exposes poses. A bridging layer must be built. Second, the map must be validated for a restaurant environment specifically — a service lane bordered by walls or barriers on both sides, with alcoves for tables. The geometric constraints of a service lane (narrow corridor, 90° turns into table alcoves, no open space for error recovery) are different from the open-plan environments in which RTAB-Map has typically been evaluated. No prior work has evaluated RTAB-Map mapping and localization accuracy in a service-lane restaurant topology.

### 2.2.3 Autonomous Navigation

Once a map is built and the robot can localize within it, the navigation stack must plan a path to a goal and execute it. The Navigation2 (Nav2) framework is the standard ROS2 navigation stack [2.2.15].

**Global planning.** The global planner computes a path from the robot's current pose to the goal pose on a static costmap — the pre-built occupancy grid map inflated by the robot's footprint. The path minimizes a cost function that penalizes proximity to obstacles and path length. Standard global planners include NavFn (a Dijkstra-based planner on a potential field) and Smac Planner (a more recent hybrid-A* planner that produces smoother, kinematically feasible paths) [2.2.16].

**Local planning and control.** The local controller executes the global path by generating velocity commands at the control loop frequency — typically 20–50 Hz. The Dynamic Window Approach (DWB) controller samples candidate velocity pairs $(V_x, V_\omega)$ within the robot's kinematic limits, scores each against a weighted cost function (goal progress, obstacle clearance, path alignment, speed), and selects the highest-scoring command [2.2.17]. The local costmap — a rolling window around the robot, updated with live sensor data — captures dynamic obstacles that were not in the static map, though in a lane-separated restaurant design the primary obstacle sources are objects that accidentally enter the lane.

**Non-holonomic constraints.** A two-wheel differential-drive robot is non-holonomic: it cannot move laterally ($V_y = 0$). Every position correction requires a sequence of rotation and translation. Nav2 supports this constraint through the kinematic model configured in the controller server — the DWB planner only samples velocity pairs that satisfy the differential-drive constraint. In-place rotation handles heading correction; Nav2's behavior tree can trigger a 90° or 180° rotate-in-place recovery when the robot fails to make progress toward the goal.

**Behavior trees and recovery.** Nav2 uses behavior trees to orchestrate the navigation lifecycle: compute a path, follow it, monitor progress, and recover from failures [2.2.18]. If the robot gets stuck — blocked by an obstacle, unable to progress along the path — the behavior tree triggers recovery actions in sequence: a small rotation to clear the obstacle, a larger in-place rotation, and finally a full replan from the current pose. If all recovery actions fail, the navigation task is aborted and the failure is reported.

**Prior work on Nav2 for service robots.** Nav2 has been deployed on ROS2 delivery robots in academic settings — campus food delivery, hospital medication transport, office document delivery [2.2.19]. These systems share a common pattern: a human operator selects a goal (a waypoint on the map), the robot navigates there, reports arrival, and waits for the next instruction. Navigation success rates exceeding 90% are reported in controlled indoor environments.

**→ Gap.** The pattern in all prior work is operator-initiated goal assignment: a human selects the target. The open problem is coupling Nav2's goal interface to an external, non-human source — an AI agent that emits navigation goals as side effects of business events. When the orchestrator marks an order as Xong (completed), it must resolve `table_id` to a goal pose and send that goal to Nav2 — without a human in the loop. When a new goal arrives while the robot is en route to a previous goal (a higher-priority delivery), Nav2 must preempt the current navigation and route to the new target. The navigation capability is mature. The integration layer — dynamic goal assignment from an AI agent, goal preemption based on business priority, and bidirectional state reporting (robot pose → backend, backend goal → Nav2) — is the specific gap that prior work has not addressed.

### 2.2.4 Fiducial Marker Docking

SLAM-based localization accumulates drift: residual map error, odometry error, and localization error combine to produce a pose estimate that can be off by 5–15 cm even under good conditions. For food delivery, this error matters at the final approach — a 10 cm lateral error means the robot does not face the table squarely, making it difficult for the customer to reach the food tray. Fiducial markers — visual patterns with known geometry — provide an absolute local reference that corrects the accumulated drift at the target.

**ArUco markers.** An ArUco marker is a square black-and-white fiducial pattern encoding a unique integer ID [2.2.20]. Detection proceeds in three stages: contour extraction from the camera image, perspective correction to produce a front-facing square, and bit-pattern decoding to recover the marker ID. The marker's physical size is known, so the 6-DoF camera-to-marker transform can be recovered via Perspective-n-Point (PnP) pose estimation: given the 3D coordinates of the marker's four corners in the marker's local frame (a 2D plane at z = 0 with known side length) and their 2D projections in the camera image, PnP solves for the camera's rotation and translation relative to the marker. The solution is unique and well-conditioned when the marker occupies a sufficient portion of the image.

**Why ArUco for table docking.** At the final approach to a table — approximately 1 meter from the expected table position — the robot's D435 camera searches for the table's assigned ArUco marker. Upon detection, the PnP-computed pose replaces the SLAM-based pose estimate at that instant, providing a correction that is independent of all prior drift. The robot then executes a short final approach — 0.5–1.0 meters — to a pre-defined docking pose relative to the marker. Because this final segment is short, the odometry error accumulated during those 0.5–1.0 meters is typically within 1–2 cm, achieving the precision needed for food delivery [2.2.21].

**Prior work on marker-based docking.** ArUco docking has been demonstrated on ROS2 robots for two primary use cases: autonomous charging (the robot navigates to a charging dock and uses an ArUco marker for precise alignment with the charging contacts) and delivery drop-off (the robot navigates to a designated drop-off point and uses a marker to position itself accurately). In both cases, each marker is treated as an independent navigation target — a pose in space that the robot must reach.

**→ Gap.** Prior work treats markers as geometric targets only. In a restaurant AI system, each marker carries a business identity: marker ID 5 is not just a pose — it is "table B3, currently occupied by session S42, which has active order #O128." The system must verify upon docking that the robot is at the correct table for the order it is carrying. A marker detected at pose $(x, y, \psi)$ does not tell the system *whose food is on the tray*. The business-context binding — marker ID → table ID → session ID → order ID — must be configured and verifiable by the backend at docking time. No prior system has implemented this semantic marker binding where the ArUco detection triggers not just a pose correction but a business-context confirmation: "robot R1 is at table B3, carrying order O128 for session S42 — correct, proceed with delivery."

Additionally, ArUco reliability under restaurant conditions — variable lighting from windows and overhead fixtures, partial occlusion when the robot approaches at an angle — has not been systematically evaluated for a per-table docking scenario with consumer-grade RGB-D cameras. Prior evaluations use controlled lighting.

### 2.2.5 Prior ROS2 Delivery Robot Research — The Interaction Gap

Academic projects have demonstrated ROS2-based delivery robots in various service contexts: campus cafeteria food delivery, hospital ward medication transport, office building document delivery [2.2.22]–[2.2.24]. The typical hardware configuration — 2D LiDAR + RGB-D camera + IMU + wheel encoders — mirrors the sensor suite described in this thesis. The typical software pipeline — RTAB-Map or Cartographer for SLAM, Nav2 for navigation, ArUco markers for precision docking — mirrors the navigation approach described in this thesis. These systems have demonstrated that autonomous indoor delivery is feasible: robots navigate from origin to destination, avoid static obstacles, and dock at target locations with acceptable precision.

The common finding is that every system surveyed handles physical movement only. The robot drives to a table and stops. There is no conversational interaction — the robot cannot take orders, answer menu questions, confirm selections, or process payment. The robot delivers food that was ordered through a separate channel; it does not participate in the ordering process. The navigation problem is solved; the interaction problem is unaddressed [2.2.25].

### → Overall Gap for §2.2

The field can build a robot that navigates to a fixed point selected by a human operator. It cannot build a robot whose navigation targets are dynamically assigned by an external AI agent managing live restaurant state, whose arrival at a table triggers an ArUco-verified business-context confirmation, and whose odometry survives repeated service cycles with cumulative drift tracking.

Specifically, four component gaps constitute the overall need:

1. **EKF odometry for service cycles.** No prior work has evaluated EKF-fused encoder+IMU odometry under repeated restaurant service cycles — kitchen→table→kitchen loops with acceleration from rest, short straight segments, and in-place rotations — using consumer-grade MPU6050-class IMU on a purchased TWD platform, where cumulative drift across multiple cycles determines ArUco re-detection success.

2. **Restaurant-specific SLAM with semantic table waypoints.** No prior work has built an RTAB-Map map of a service-lane restaurant topology and exposed table waypoints as queryable navigation targets to an external backend — or validated localization accuracy in the geometrically constrained environment of a service lane bounded by walls on both sides.

3. **Dynamic goal coupling with an external AI agent.** No prior work has connected a Nav2 navigation stack to a goal source that is external, AI-driven, and dynamic — where goals arrive asynchronously, preempt in-flight navigation, and carry business context (order ID, session ID) that must be reported back upon arrival.

4. **Business-context ArUco docking.** No prior work has bound ArUco markers to restaurant business entities — table ID, session ID, order ID — so that marker detection at docking time triggers a semantic verification ("robot is at the correct table for this order") rather than only a geometric pose correction.

These four gaps motivate the system requirements for autonomous navigation in §3.1 and the proposed method in §3.4–§3.7.
