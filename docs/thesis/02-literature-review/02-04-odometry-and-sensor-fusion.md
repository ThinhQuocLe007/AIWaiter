# 2.4 Odometry and Sensor Fusion

For a mobile robot to follow a path or return to a point, it has to keep track of where it is. One basic way to do this is odometry: the robot starts from a known position and orientation, and then keeps adding up its own small movements over time to work out its current pose. Because it builds the estimate step by step from the robot's own motion, without any outside reference, odometry is a form of dead-reckoning.

Odometry can be computed from different sensors, and each sensor gives its name to a type of odometry. Wheel odometry uses the wheel encoders to measure how far each wheel has turned. Inertial odometry uses an IMU, adding up its measured accelerations and rotation rates. Visual, or visual-inertial, odometry tracks how the image from a camera shifts between frames, sometimes together with an IMU. Laser odometry matches one LiDAR scan to the next to estimate how the robot has moved between them. This work uses wheel odometry as its primary source of translational motion, since the robot has wheel encoders and moves on a flat indoor floor where wheel motion is a good indicator of travel. It is not used alone, however: the wheel odometry is fused with the IMU through an Extended Kalman Filter, so that the encoder supplies the forward motion while the IMU corrects the heading.

Wheel odometry is attractive because it is simple, runs at a high rate, needs no external infrastructure, and gives a smooth estimate of motion. Its weakness comes from the very way it works: because it only ever adds new movements onto the previous estimate, it has no way to correct a mistake once one has been made. Small errors - from a wheel briefly slipping, from tyres that are not exactly the assumed size, or from rounding in the computation - are added in and kept forever. Over a long run these errors accumulate, so the estimated position slowly and steadily moves away from the true position. This effect is called drift, and it grows without bound the longer the robot travels.

Every type of odometry drifts in some way, but they do not all drift for the same reasons: wheel odometry suffers mainly from slip, while an IMU's heading drifts from the build-up of small measurement errors. Because their errors are of different kinds, a single, more reliable estimate can be obtained by combining several sensors instead of trusting any one of them - this is called sensor fusion. The encoder and the IMU make a good pair to fuse: the encoder gives accurate motion over short distances but drifts with slip, while the IMU captures fast turns well and does not depend on the wheels gripping the floor, but its heading drifts when its readings are added up. Each one is strong exactly where the other is weak.

A common and effective way to fuse such sensors is the Kalman filter. Rather than simply averaging the sensors, it keeps a running estimate of the robot's state - its pose and velocity - together with a measure of how uncertain that estimate is. It then repeats two steps. In the prediction step, it advances the estimate using a model of how the robot moves. In the update step, it corrects the estimate each time a new sensor reading arrives, giving more weight to whichever source it currently trusts more, according to how uncertain each source is. In this way the filter naturally leans on the encoder for smooth short-term motion and on the IMU for reliable turning, so the fused result is better than either sensor on its own.

The motion of a wheeled robot, and the way its sensors relate to its state, are not linear - they involve the robot's heading through sine and cosine terms - so the basic Kalman filter, which assumes linear relationships, cannot be applied directly. The extended Kalman filter (EKF) deals with this by approximating these non-linear relationships with a linear one at each step, around the current estimate, which lets the same predict-and-update idea be used on a non-linear system like this robot.
