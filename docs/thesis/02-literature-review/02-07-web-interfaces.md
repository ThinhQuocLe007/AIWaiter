---

Restaurant operations require distinct user interfaces for each operational role. A customer seated at a table needs a menu browsing and ordering tablet with voice conversation display and cart management. Kitchen staff need a real-time order board showing pending, in-progress, and completed dishes. A front-of-house manager needs a fleet dashboard with live robot positions and table status. A guest arriving at the entrance needs a check-in kiosk to select a table and confirm seating. All four interfaces share a single source of truth — the backend orchestrator's restaurant state — and all must update in real time as that state changes, driven by AI agent events: orders created, cart items modified, robots dispatched, payments processed.

This section surveys the technology foundations for building such interfaces: single-page application (SPA) frameworks, component libraries, build tooling, and real-time communication patterns. Each sub-section surveys available options and identifies selection criteria for the specific context of a multi-role, AI-driven restaurant system — a context that prior web technology surveys have not addressed.

### 2.7.1 Single-Page Application Frameworks

The SPA model — a single HTML page with client-side routing, where reactive component trees update in place as data changes — is the standard modern architecture for real-time dashboards, interactive ordering systems, and operational panels. Three frameworks dominate the landscape.

**Vue 3 (Composition API, TypeScript).** Vue 3 provides a reactivity system based on JavaScript Proxies: component state declared via `ref()` and `reactive()` automatically triggers DOM updates when modified [2.7.1]. The Single-File Component (SFC) format separates template (HTML), logic (TypeScript), and style (CSS) within one file — a structure well-suited to restaurant interfaces where each component (menu card, order item, kitchen ticket, robot card) has a distinct visual representation. Pinia provides cross-component state management with devtools integration [2.7.2]. Vue Router enables client-side navigation without full-page reloads. First-class TypeScript support means that the shared client library can import type definitions that mirror the backend's Pydantic schemas — a `Dish` type, an `Order` type, a `RobotState` type — ensuring type safety across the API boundary. Vue's runtime is approximately 33 KB gzipped [2.7.3]. Vietnamese character rendering requires no additional configuration — Vue's template compiler emits standard HTML that the browser renders correctly for Unicode Vietnamese text.

**React (Hooks, JSX).** React is the dominant SPA framework by market share and ecosystem size [2.7.4]. Its component model uses JSX — an HTML-in-JavaScript syntax — and hooks (`useState`, `useEffect`) for state and side-effect management. State management options include Context API (built-in), Redux, and Zustand. The ecosystem includes Next.js for server-side rendering, Material UI and Ant Design for component libraries, and React Query for server state synchronization. React's strength is its ecosystem maturity and the availability of third-party libraries for virtually any UI pattern. Its trade-offs for the restaurant context are: (a) JSX blends markup and logic in a way that can obscure template structure in complex multi-form interfaces (order forms with validation, kitchen boards with drag-and-drop), (b) the hooks model introduces subtle correctness pitfalls — stale closures in `useEffect` dependencies, missing cleanup functions for WebSocket subscriptions — that add debugging overhead, and (c) the runtime is approximately 42 KB gzipped (React + ReactDOM) before additional state management [2.7.5].

**Angular (TypeScript, RxJS).** Angular is an opinionated full framework with dependency injection, RxJS observables for asynchronous state streams, and a module-based architecture [2.7.6]. Strong typing and a structured project layout make it well-suited to large enterprise teams with established conventions. Its trade-off for the restaurant context is overhead: the runtime is significantly larger than Vue or React, the learning curve is steep, and the boilerplate required for simple components is disproportionate for interfaces where business logic lives on the backend server, not in the browser. Angular's RxJS observable model handles WebSocket event streams naturally, but the same pattern is achievable in Vue with a simple composable and in React with a custom hook — without the framework-level complexity.

**Comparison.** The following table summarizes the three frameworks against criteria relevant to a multi-role, AI-driven restaurant interface:

| Criterion | Vue 3 | React | Angular |
|-----------|:-----:|:-----:|:-------:|
| Reactivity model | Proxy-based (`ref`, `reactive`) | Virtual DOM + hooks | Zone.js + RxJS |
| TypeScript support | First-class | Good | First-class |
| Runtime size (gzipped) | ~33 KB | ~42 KB | ~65+ KB |
| Vietnamese text rendering | Native (Unicode) | Native (Unicode) | Native (Unicode) |
| SFC separation (HTML/CSS/TS) | ✓ (built-in) | ✗ (JSX) | ✓ (built-in) |
| Multi-app monorepo support | ✓ (Vite workspaces) | ✓ (Nx, Turborepo) | ✓ (Nx) |
| Developer ramp-up time | Low | Medium | High |
| Real-time WS integration | Composable pattern | Custom hook | RxJS Observable |

No framework is inherently unsuitable. The choice is determined by the specific requirements of the project: three independent SPAs sharing a common TypeScript client library, with reactive Vietnamese text rendering, WebSocket-driven UI state, and a low ramp-up time for a small team. Prior work has compared these frameworks for general web development [2.7.7]; no survey has established selection criteria for the specific context of a multi-role, AI-driven restaurant system where the event source is an external agent rather than a human user.

**→ Gap:** The framework selection for a restaurant AI system is not evaluated in prior literature. The specific combination of requirements — (a) multiple role-specific SPAs sharing TypeScript types with the backend, (b) real-time UI state driven by WebSocket events from an AI agent, (c) reactive Vietnamese text rendering for conversation transcripts and dish names, and (d) a small-team development context — establishes criteria that general web framework comparisons do not capture.

### 2.7.2 Component Libraries and Build Tooling

**Component libraries.** Restaurant interfaces require data-intensive, touch-friendly components. A menu browsing interface needs a scrollable, filterable data table with image thumbnails and price columns. A kitchen Kanban needs draggable card components with status badges and elapsed-time indicators. A fleet dashboard needs status cards with battery gauges and live position markers. A check-in kiosk needs a table grid with real-time occupancy indicators and a party-size selector. The following component libraries target the Vue 3 ecosystem:

- **PrimeVue 4** [2.7.8]: provides DataTable with sorting, filtering, pagination, and row expansion — mapping directly to menu browsing and order history views. Form components (`InputNumber`, `Dropdown`, `Textarea`) with built-in validation. Card and Panel containers for layout. Dialog and OverlayPanel for payment modals and confirmation prompts. Badge and Tag for order status indicators. Toast for notifications. Full TypeScript support with typed component props and events. Tree-shakeable imports minimize per-app bundle size. The library's design language is neutral — it does not impose a visual identity — which suits a multi-role system where the customer tablet, kitchen panel, and management dashboard need distinct visual treatments.

- **Vuetify 3** [2.7.9]: Material Design component library. Provides a comprehensive set of components with a consistent visual language. Well-suited to admin dashboards and data-heavy panels. Trade-off: Material Design's rigid grid system and elevation-based layering constrain the responsive, touch-friendly menu browsing layout that a customer-facing restaurant tablet requires. Heavier bundle weight than PrimeVue due to the full Material Design specification implementation.

- **Ant Design Vue** [2.7.10]: enterprise-grade component library with comprehensive form, table, and layout components. Strong for data management interfaces (kitchen panel, fleet dashboard). Trade-off: visual style optimized for enterprise back-office applications — dense forms, compact tables, subdued colors — which does not match the warm, inviting visual language expected in a customer-facing restaurant interface. Larger bundle size and more complex component API.

**Build tooling.** Vite is the standard build tool for Vue 3 projects as of 2024 [2.7.11]. Its native ES module dev server provides hot module replacement in under 50 milliseconds regardless of project size — relevant for a three-application monorepo where each SPA must be independently developed and served during development. Production builds use Rollup with tree-shaking and code-splitting. Vite's plugin ecosystem supports TypeScript, Vue SFC compilation, and CSS pre-processing natively. The alternative, Webpack via Vue CLI, has been largely superseded in the Vue ecosystem due to slower dev server startup and HMR on large projects. For a three-app monorepo where developers frequently switch between the customer tablet, kitchen panel, and kiosk during development, fast HMR directly impacts iteration speed.

**Prior work:** Component library comparisons exist for general web development contexts [2.7.12]. Build tool benchmarks exist comparing Vite and Webpack [2.7.13]. No prior evaluation has assessed component libraries or build tooling against the specific requirements of a multi-role restaurant interface: Vietnamese text rendering with diacritic accuracy, touch-friendly tablet interfaces with large tap targets suitable for restaurant environments, real-time data binding to WebSocket events from a backend orchestrator, and a multi-app monorepo structure where three SPAs share a common client library.

**→ Gap:** Neither component library selection criteria nor build tooling evaluation has been conducted for the specific context of AI-driven restaurant interfaces. The requirements — Vietnamese diacritic rendering, touch-first tablet interfaces, WebSocket-driven data tables and status indicators, and a multi-app monorepo build pipeline — define a selection context that prior general-purpose surveys do not address.

### 2.7.3 Real-Time Communication Patterns for Multi-Role UIs

Restaurant interfaces cannot rely on polling-based data refresh. When an AI agent creates an order, the kitchen display must show it immediately — not on the next 5–10 second poll cycle. When the agent modifies a cart, the customer tablet must reflect the change before the next utterance. Three communication patterns exist for real-time web applications:

**Polling (REST-based periodic refresh).** The client sends an HTTP GET request every N seconds to check for state changes. This is the standard pattern in traditional restaurant POS and KDS systems, where a human operator glances at the screen periodically and a 5–10 second delay is acceptable [2.7.14]. For a voice-driven customer interaction, a 5-second delay between the utterance "Cho 2 Ốc Hương" and seeing the cart update on the tablet breaks conversational flow — the customer will repeat the order or lose confidence that the system understood. Polling also wastes bandwidth: the tablet polling `/orders/{table_id}` 12 times per minute, regardless of whether any order exists. For a display panel with 10-second poll cycles, the average latency from event occurrence to UI update is 5 seconds, with worst-case 10 seconds.

**WebSocket (full-duplex persistent connection).** A single TCP connection is upgraded to a WebSocket, providing bidirectional communication for the lifetime of the session [2.7.15]. The server pushes events to clients as they occur: `order.created`, `cart.updated`, `robot.position`, `voice.reply`. Role-based fan-out routes each event to the correct client subset — the kitchen panel receives `order.created` but not `voice.reply`; the customer tablet receives `voice.reply` and `cart.updated` but not `task.assign`; the robot receives `task.assign`; the fleet dashboard receives `robot.updated`. Event delivery latency is sub-second after the state change occurs on the server. Auto-reconnection with exponential backoff (1s → 2s → 4s → … cap 30s) handles temporary WiFi drops — the standard resilience pattern for production WebSocket deployments [2.7.16].

**Server-Sent Events (SSE).** SSE provides server-to-client streaming over a standard HTTP connection [2.7.17]. The server opens a long-lived HTTP response and writes events as they occur; the browser's `EventSource` API receives them. SSE is lighter-weight than WebSocket for unidirectional server→client traffic and reconnects automatically. It is used by the agent brain to stream LLM-generated responses sentence-by-sentence to the voice pipeline and tablet — conversational text that arrives incrementally rather than all at once. SSE is not suitable for bidirectional communication: robot telemetry (robot → server), tablet commands (tablet → server to arm microphone), and kiosk seating requests (kiosk → server) require the client to send data, which SSE does not support.

**Prior work on real-time restaurant systems:** Restaurant management platforms implement real-time state propagation internally using proprietary protocols [2.7.18]. These protocols are not documented publicly and provide no API for third-party integration — an external AI agent cannot subscribe to `order.created` events. Academic work on real-time multi-role web systems exists for domains outside restaurants: hospital patient monitoring dashboards use WebSocket for vitals streaming [2.7.19]; logistics control panels use WebSocket for vehicle position tracking [2.7.20]; financial trading UIs use WebSocket for price feeds. No prior system provides a documented WebSocket event catalog for restaurant operations where the event source is an AI agent, or evaluates which real-time pattern (polling vs. WebSocket vs. SSE vs. hybrid) is appropriate for each restaurant role and event type.

**→ Gap:** The combination of WebSocket for bidirectional multi-role state synchronization and SSE for AI agent response streaming, with a documented event catalog specific to restaurant operations driven by an AI agent, has not been described in prior literature. The selection criteria — which roles need bidirectional communication (customer tablet, robot), which need server→client only (kitchen panel, fleet dashboard), and which benefit from streaming (agent responses) — establish a hybrid architecture that general real-time web surveys do not address.

### 2.7.4 Multi-Role SPA Architecture

The multi-role SPA pattern — multiple single-page applications, each serving one user role with role-specific UI and event subscriptions, sharing a common client library — is the standard architecture for systems where different user types need different views of the same underlying data. It has been documented for enterprise SaaS platforms (admin dashboard vs. customer portal vs. agent console) and for operational systems (dispatcher view vs. field-worker view vs. supervisor view). The pattern is well-understood in general web architecture literature.

What has not been documented is this pattern applied to a restaurant system where the shared state is driven by an AI agent. In a conventional multi-role SPA, the event source is typically a human user: an admin updates a record → other roles see the update. In a restaurant AI system, the event source is the agent: the agent confirms an order → the kitchen panel shows a new ticket and the customer tablet shows the confirmed cart. The agent dispatches a robot → the fleet dashboard shows the robot's status change and the robot receives a navigation goal. The agent processes payment → the session closes and the table status updates. All roles are downstream of the agent's decisions, and the real-time synchronization must guarantee that no role sees a state that has already been superseded by the agent's next action.

The additional constraint specific to this context is the shared TypeScript client library: the three SPAs must import types — `Dish`, `Order`, `CartItem`, `RobotState`, `TableStatus` — that mirror the backend's Pydantic schemas, ensuring that a type mismatch between the orchestrator API and any frontend is caught at compile time. This pattern — shared types across a Python backend and TypeScript frontends — is established in the broader web development literature (OpenAPI code generation, tRPC for TypeScript-only stacks) but has not been evaluated for the Python-FastAPI-to-Vue-TypeScript bridge that this project requires.

**→ Overall gap for §2.7:** No prior work establishes: (a) SPA framework selection criteria for a multi-role, AI-driven restaurant system — reactivity and TypeScript requirements, Vietnamese text rendering, small-team ramp-up time, and multi-app monorepo structure, (b) component library evaluation against restaurant-specific UI requirements — Vietnamese diacritic rendering, touch-friendly tablet interfaces, data-intensive real-time displays — with a documented selection rationale, (c) a hybrid real-time communication architecture — WebSocket for bidirectional multi-role state synchronization, SSE for AI agent response streaming — with a per-role, per-event-type rationale, and (d) a multi-role SPA architecture where the shared TypeScript client library mirrors Pydantic backend schemas and all roles are downstream of an AI agent's business events. The individual technologies — Vue 3, PrimeVue, Vite, WebSocket, SSE — are individually mature. Their composition into a documented multi-role restaurant interface architecture, and the selection criteria that justify that composition, has not been described in the literature. This gap motivates the web interface architecture in §4.8.
