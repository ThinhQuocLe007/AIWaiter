## 2.4 Conversational AI Agent

> *A restaurant agent must understand Vietnamese speech, decide on an action, and execute that action correctly — without hallucinating dish names or violating the order process. This section surveys the components of task-oriented conversational agents: agent architectures for orchestrating LLM-tool interaction (§2.4.2), the LLM reasoning engine (§2.4.3), intent classification to determine which tool an utterance demands (§2.4.4), action validation to prevent hallucinated arguments from reaching external tools (§2.4.5), memory and state management (§2.4.6), and planning, tool composition, and domain adaptation (§2.4.7).*

### 2.4.1 From General-Purpose LLM to Task-Oriented Agent

The Transformer architecture, introduced by Vaswani et al. [2.4.1], replaced recurrent sequence processing with self-attention, enabling models to capture long-range dependencies without the sequential bottleneck of LSTMs. Scaling these architectures along parameter count, training data volume, and compute produced large language models (LLMs) whose performance on language tasks improved in a predictable power-law relationship [2.4.2]. These models, trained on next-token prediction over web-scale corpora, exhibited emergent capabilities including translation, summarization, reasoning, and instruction following [2.4.3].

Despite their generative capabilities, LLMs remain fundamentally text-in-text-out systems. They cannot modify a database, update application state, or command external devices — they map input tokens to output tokens, and those output tokens sit in a buffer unless acted upon by external code. The earliest approach to bridging this gap was post-hoc parsing: an LLM produced free-text instructions such as "add Ốc Hương to the cart," and a downstream parser — typically regex patterns, keyword matching, or a separately trained classifier — extracted the structured intent [2.4.4]. This pipeline was brittle because the parser had to anticipate every possible phrasing the LLM might produce, and any variation broke the extraction.

Function calling [2.4.5] resolved this fragility by making structured invocation a first-class capability of the LLM API itself. Rather than relying on a downstream parser, the LLM is given a schema defining available tools — each with a name, natural-language description, and typed JSON Schema for its parameters — and produces a structured invocation object such as `{"name": "search", "arguments": {"query": "Ốc Hương"}}`. The serving framework intercepts this output, validates its syntactic correctness, and executes the corresponding function against live systems, feeding the return value back to the LLM for the next reasoning step. Introduced by OpenAI and subsequently adopted across Anthropic, Google, and Ollama [2.4.6], function calling is now the standard mechanism for converting LLM output into system actions. Figure 2.7 illustrates this mechanism.

**[Figure 2.7 — Function calling mechanism: the LLM receives a tool schema alongside the conversation, produces a structured JSON invocation rather than free-text description, and the framework executes the tool and feeds results back.]**

Function calling provides the mechanism for action, but it does not guarantee correctness. An LLM can invoke the correct tool with hallucinated arguments, call tools in an invalid sequence, or produce parameter values that violate domain constraints. The layers surrounding the LLM — the orchestration architecture, routing mechanism, validation gates, memory system, and planning logic — determine whether tool invocations are safe for transactional domains where errors have material consequences. Each layer is surveyed in the subsections that follow.

### 2.4.2 Agent Architectures — The Orchestration Layer

The architecture surrounding the LLM determines when tools can be invoked, what happens between invocations, and what guarantees the system provides about correctness and termination. Four architectural patterns have been studied in the agent development literature, each representing a different approach to governing LLM-tool interaction.

The simplest pattern is the chain, in which the developer declares a fixed linear sequence of steps — classify the utterance, extract parameters, call a tool, generate a response — with no branching. As implemented in LangChain's LCEL [2.4.7], each utterance type follows its own predetermined chain, and the control flow is explicit, reproducible, and debuggable. Prior work on chain-based task-oriented dialogue [2.4.8] has demonstrated reliable execution for narrow domains with predefined workflows. The chain's primary strength is deterministic governance: every step, transition, and termination condition is hard-coded by the developer, and the execution trace is fully auditable. Its primary limitation is rigidity: a routing error at step one is unrecoverable because no architectural component inspects the utterance mid-chain and redirects. Error recovery requires enumerating every possible failure mode at every step, a combinatorial problem when the input space is open-ended natural language.

To address the chain's inability to adapt to conversational variation, the autonomous reasoning loop — introduced by the ReAct pattern [2.4.9] — delegates governance to the LLM itself. On each iteration, the model receives the current state (conversation history, tool definitions, and prior tool results) and produces either a reasoning trace, a tool call, or a final answer. The framework executes any tool call, feeds the result back into the loop, and the next iteration begins. AutoGPT [2.4.10] extended this pattern with internet access, file system operations, and long-term memory, enabling agents to pursue open-ended goals across dozens of iterations without human intervention. The loop's strength is conversational flexibility: the LLM adapts its next action based on what it observes, without requiring the developer to enumerate dialogue paths.

The loop's flexibility is achieved by relinquishing the deterministic governance that chains provide. Termination is not guaranteed: the loop ends when the LLM decides to stop, and if the model enters a reasoning cycle — producing neither a tool call nor a final answer — the turn blocks indefinitely [2.4.11]. The architecture provides no structural mechanism to enforce a valid tool sequence or to validate tool arguments before execution. ReAct has been evaluated on knowledge-intensive benchmarks such as HotpotQA and FEVER [2.4.9], where it improved task completion rates over chain-based baselines. Published limitations note that the architecture cannot guarantee bounded execution or action correctness because governance and proposal reside in the same probabilistic component.

Graph-based architectures, as implemented in LangGraph [2.4.12], restore deterministic governance by encoding the state machine in the architecture's topology rather than in prompt instructions. The developer declares a directed graph: a typed, shared state object flows through nodes (LLM calls, tool executions, or deterministic functions) connected by conditional edges that inspect the state and determine which node executes next. Unlike a chain, the graph can branch based on runtime conditions. Unlike a loop, termination is structural — guaranteed by the graph topology, not emergent from the LLM. The architecture provides three primitives: a checkpointer that serializes state after each node execution for session-scoped persistence [2.4.13]; conditional edges that enable mandatory validator nodes on every path between LLM calls and tool executions; and circuit breakers that enforce bounded execution by capping retry iterations before routing to a fallback response. LangGraph has been evaluated primarily on code generation and multi-step reasoning tasks [2.4.12]; its applicability to task-oriented dialogue in Vietnamese has not been assessed.

Multi-agent architectures — AutoGen [2.4.14], CrewAI [2.4.15], and CAMEL [2.4.16] — decompose the single-agent model by distributing tools, prompts, and responsibilities across independent LLM instances that communicate through structured messages. AutoGen supports hierarchical delegation where a manager LLM routes tasks to specialized workers, and CAMEL introduced role-playing LLM pairs for task simulation and synthetic data generation. The defining strength is specialization: each agent operates with a narrower scope and a more focused prompt than a single agent handling all tools, which the literature suggests should produce more reliable behavior within each agent's bounded domain [2.4.14]. Three documented limitations emerge from this decomposition. First, attention dilution [2.4.17]: as agents exchange messages, each agent's context window accumulates not only the user utterance but every peer agent's reasoning traces, tool outputs, and coordination instructions, and benchmarks document accuracy regression when agent count exceeds two to three. Second, coordination is entirely LLM-mediated — the manager invokes an LLM to decide which worker to route to, and workers invoke LLM calls to hand off to peers — so there is no deterministic coordination layer. Third, business process ownership is diffused across agents: no single component is structurally responsible for the correctness of the overall tool sequence.

No architecture has been evaluated for task-oriented dialogue in Vietnamese, where conversational flexibility and deterministic process enforcement are simultaneously relevant. Chain architectures have been validated on narrow-domain task-oriented dialogue [2.4.8] but not on the open-ended variability of Vietnamese restaurant speech. Loop architectures have been assessed on English knowledge benchmarks [2.4.9] but not on transactional domains where action validation and bounded execution are correctness requirements. Graph-based architectures provide primitives for combining deterministic governance with LLM flexibility — conditional edges for mandatory validation, circuit breakers for bounded execution, checkpointers for session persistence — but their application to a domain where every LLM-proposed action must be inspected before execution has not been studied. Multi-agent architectures have been evaluated on cooperative reasoning tasks [2.4.14] but not on Vietnamese dialogue, and the documented attention dilution problem [2.4.17] has not been characterized for task-oriented agent coordination.

### 2.4.3 Large Language Models — The Reasoning Component

The LLM is the component that processes user input, conversation history, and system prompts to decide which tool to call and with what arguments. Three categories of Vietnamese-capable LLMs have been documented in the literature, each making a different trade-off among language quality, function-calling capability, and deployment model. Beyond these dimensions, a model's context window determines the maximum input length it can process in a single forward pass, and its serving infrastructure determines how it is deployed and at what cost.

The most natural approach to Vietnamese dialogue is to use a model trained specifically on Vietnamese. PhoGPT (7B) [2.4.18] is a decoder-only Transformer trained from scratch on a Vietnamese-dominant corpus. It has been evaluated on Vietnamese-specific benchmarks including VLSP [2.4.19], where it demonstrates grammatically fluent output with correct tonal diacritics and appropriate politeness particles (dạ, ạ) that multilingual models often flatten or omit. ViSoBERT [2.4.20], an encoder-only model based on BERT, targets classification and embedding tasks rather than text generation. The defining strength of Vietnamese-specific models is language quality: they are native Vietnamese speakers in a way that multilingual models, where Vietnamese is one of dozens of languages competing for representation, cannot match. Their defining limitation is the absence of function calling as a first-class API capability. As of 2025, neither PhoGPT nor ViSoBERT can produce the structured `{"name": "...", "arguments": {...}}` invocation format that agent frameworks intercept and execute. They can describe what action should be taken in free text, but deploying them in a task-oriented agent would require reverting to the post-hoc parsing architecture described in §2.4.1.

To obtain function calling while retaining the ability to self-host, open-weight multilingual models offer a different trade-off: they sacrifice Vietnamese-specific optimization for ecosystem maturity. The Qwen2.5 series (0.5B to 72B parameters) [2.4.21] emphasizes Southeast Asian languages in its training data composition and has been evaluated on multilingual benchmarks demonstrating Vietnamese conversational capability. Llama 3 (8B to 405B) [2.4.22] includes approximately 0.5% Vietnamese text in its 15-trillion-token training corpus. Gemma 2 (2B to 27B) [2.4.23] supports Vietnamese as part of its multilingual training. All three families expose function calling as a documented API feature, assessed through the Berkeley Function Calling Leaderboard (BFCL) [2.4.24], a standardized benchmark measuring structured tool invocation accuracy across model families. Reported results separate the families consistently at the 7–9B scale, and the ordering is stable across leaderboard revisions even though the absolute figures are not: the leaderboard's scoring has changed across versions, so any specific percentage must be quoted against the version that produced it. The trade-off these models make is Vietnamese quality: they produce correct but sometimes stilted Vietnamese, with occasional diacritic errors and a tendency toward neutral rather than context-appropriate politeness registers.

At the top of the accuracy spectrum, commercial API models — GPT-4o, Claude 3.5 Sonnet, and Gemini 2.0 Flash [2.4.25]–[2.4.27] — achieve both the highest documented Vietnamese conversational quality and the highest reported BFCL results. Their strength is that they satisfy language quality and function-calling accuracy simultaneously, without the trade-off that open-weight multilingual models make. Their limitation is architectural: the deployment model requires internet connectivity for every inference call, incurs per-token API costs that accumulate with usage volume, and transmits user data to external servers outside the operator's control.

Beyond tool calling and language quality, a model's context window determines the maximum length of input it can process in a single forward pass. Context window sizes vary by an order of magnitude across model categories. At the low end, PhoGPT provides a 4,096-token window [2.4.18] — sufficient for single-turn exchanges but documented as insufficient for extended multi-turn dialogue [2.4.19]. In the mid-range, open-weight multilingual models span 8,192 tokens (Llama 3 8B, Gemma 2 9B) to 32,768 tokens (Qwen2.5 7B), with larger variants extending to 128K [2.4.21]–[2.4.23]. At the high end, commercial API models offer 128K to 1M tokens [2.4.25]–[2.4.27]. The practical significance of these differences is bounded by two documented findings. Liu et al. [2.4.28] demonstrated that model performance degrades on information positioned in the middle of a long context regardless of total capacity — the "lost in the middle" phenomenon — and Xiao et al. [2.4.29] showed that transformer attention favors tokens at context boundaries. These findings apply across all model sizes and categories. An additional factor specific to Vietnamese is token consumption. Multilingual subword tokenizers are fitted predominantly on high-resource languages, and Vietnamese diacritics and compound words are consequently segmented into more tokens than their English equivalents — so a context window of a given nominal size holds less Vietnamese conversation than the figure suggests. The size of that penalty is tokenizer-specific and is not reported for the models surveyed here [2.4.61]. Table 2.4c summarizes the documented context window capacities.

For open-weight models specifically, three serving frameworks appear in the deployment literature. Ollama [2.4.30] provides a local HTTP server with an OpenAI-compatible API, designed for single-GPU deployments with model persistence in GPU memory between requests. vLLM [2.4.31] targets higher-throughput scenarios through PagedAttention, an efficient key-value cache management scheme that enables multiple concurrent inference requests to share GPU memory without the overhead of separate model instances. llama.cpp [2.4.32] takes a different approach: GGUF quantization at 4–8 bit precision enables CPU-first inference with optional GPU offloading, trading output quality for reduced VRAM requirements. This quality trade-off has been documented to disproportionately affect languages underrepresented in training data [2.4.33], a finding particularly relevant to Vietnamese, where tonal diacritic distinctions are the fine-grained signal most vulnerable to quantization noise.


The three documented properties — function-calling support, context window capacity, and Vietnamese language quality — have been evaluated independently across model families but never jointly for a single application domain. The BFCL [2.4.24] measures tool-calling accuracy on general-domain function sets in English; no equivalent benchmark exists for domain-specific tool invocation in Vietnamese, where tool arguments include compound dish names with tonal diacritics and informal variants. Vietnamese-specific models such as PhoGPT have been evaluated on NLU benchmarks (VLSP, translation, summarization) [2.4.19] but not on function-calling tasks. Context window evaluations (Liu et al. [2.4.28], Xiao et al. [2.4.29]) characterize attention behavior on English and multilingual corpora but do not account for the higher token consumption of Vietnamese text.

### 2.4.4 Intent Classification — The Routing Layer

Before an agent can invoke a tool, it must determine which tool the customer's utterance demands. A routing error — sending a menu search utterance to the ordering subsystem — produces irrelevant results at best and incorrect system actions at worst. Five approaches to intent routing have been studied in the task-oriented dialogue literature.

The earliest approach, deployed in platforms such as Rasa [2.4.34] and Dialogflow [2.4.35], uses supervised classifiers — keyword matching, SVM with n-gram features, or LSTM-based sequence taggers — trained on labeled utterance-intent pairs. These classifiers are fast (~5ms) and deterministic, and have been deployed at scale for English, Chinese, Korean, and Japanese restaurant ordering. Each classifier is language-specific: keyword lists and n-gram features must be manually curated, and informal variants such as teencode abbreviations are out-of-vocabulary for models trained on formal corpora. A further limitation is the absence of conversation state: text features alone cannot distinguish a confirmation utterance from a greeting, because both produce identical n-gram vectors.

Lightweight neural classifiers, including fastText [2.4.36] and SetFit [2.4.37], improve on classical approaches by using subword n-gram features and few-shot fine-tuning of sentence transformers. Subword features provide partial robustness to previously unseen words, and inference remains fast (2–10ms) and deterministic. These classifiers share the limitation of text-only approaches: they process each utterance in isolation without access to prior turns or dialogue state.

Semantic centroid routing [2.4.38] encodes each intent as the mean embedding of several labeled example utterances and classifies new inputs by cosine similarity to these centroids. Because the same embedding model processes both training examples and user utterances, new vocabulary items are encoded in the same space without retraining. Inference is approximately 15ms after embedding. Published accuracy on domain-specific classification benchmarks ranges from 75–90% [2.4.39], with failure concentrated in three patterns: informal variants produce embeddings distant from their formal centroids, context-dependent utterances are approximately equidistant from multiple centroids, and multi-intent utterances produce a blended embedding matching neither.

LLM-based routing [2.4.40] delegates classification to the LLM by prompting it with the utterance, conversation history, and a list of intents. Because the model has encountered informal Vietnamese during pretraining and can attend to prior turns injected into the prompt, it handles teencode, context-dependent ambiguity, and multi-intent compounding — the three patterns that centroid routing fails on. The documented cost is latency and non-determinism: a single classification call takes 1.5 to 2.0 seconds on consumer GPU hardware, and temperature sampling means the same utterance may produce different routing decisions across calls. A less explored variation, documented in prior work on utterance rewriting [2.4.41], uses the LLM only to decompose complex or ambiguous utterances into simpler fragments, while a separate, faster classifier handles the actual intent assignment per fragment. This decomposition-only use of the LLM has been demonstrated on English multi-intent utterances but has not been evaluated for Vietnamese, where sentence boundaries are less clearly marked and teencode compounding creates ambiguous segmentation points.

Each of the five approaches addresses a subset of the routing requirements at the cost of another. Keyword-based and lightweight neural classifiers provide speed and determinism but fail on informal language, conversation context, and multi-intent utterances. Semantic centroid routing handles domain vocabulary through shared embedding spaces but breaks on the same three patterns — teencode, context-dependent ambiguity, and multi-intent compounding — that define Vietnamese restaurant speech. LLM-based routing resolves all three patterns but introduces latency of 1.5 to 2.0 seconds per call and non-deterministic output from temperature sampling. The decomposition-only variant of LLM routing [2.4.41] reduces the LLM's role to utterance splitting, preserving the speed of a separate classifier, but has not been evaluated on Vietnamese where teencode compounds create ambiguous segmentation boundaries. State-augmented classification [2.4.42] adds dialogue context to a fast classifier but has only been assessed on English benchmarks and requires a labeled Vietnamese dataset with state annotations that does not exist.

Across all five, the same trade recurs: the mechanisms that are fast and deterministic are precisely those blind to informality, dialogue context, and multi-intent structure, and the mechanism that handles all three is the one that is slow and non-deterministic. The surveyed work treats this as a property of the choice itself, positioning the approaches along a single axis running from cheap and brittle to expensive and flexible. What it does not ask is whether the two halves of that trade are separable — whether the cases that defeat a classifier are frequent enough, or distinguishable enough in advance, to warrant a different mechanism from the cases that do not. Nor could the question be settled from the existing evidence for Vietnamese, since compound dish names, tonal diacritics, context-dependent affirmations, and teencode abbreviations each compound a different one of the five documented limitations, and no evaluation of any approach exists on speech carrying those properties.

### 2.4.5 Action Validation — The Safety Layer

Correct routing ensures the right tool is called. It does not ensure the tool is called with correct arguments. An LLM that routes an utterance to the ordering subsystem may still produce `add_cart(name="Pizza Hải Sản", quantity=1)` — a call that is syntactically valid JSON but semantically wrong because the named dish does not exist on the menu. Function calling guarantees syntactic validity; semantic correctness requires a separate mechanism. Three approaches to mitigating argument-level errors have been studied, each operating at a different point in the generation pipeline [2.4.42].

The earliest and simplest defense is constrained decoding [2.4.43], which enforces JSON schema compliance at the token generation level: the serving infrastructure restricts the LLM's output vocabulary at each generation step to tokens consistent with the target schema. This guarantees that field names, types, required fields, and value constraints match the tool definition. The guarantee is syntactic only: `{"name": "Pizza Hải Sản", "quantity": 1}` passes every schema constraint because `name` is a valid string and `quantity` is a valid integer, but neither value has been checked against any external source of truth. Constrained decoding can prevent the LLM from producing a malformed tool call; it cannot prevent it from producing a well-formed call with wrong arguments.

To address the semantic gap that constrained decoding leaves open, retrieval-augmented generation grounding [2.4.44] injects the authoritative data — for a restaurant, the actual menu — into the LLM's prompt before it generates tool arguments. If the prompt contains valid dish names and prices, the model is more likely to reproduce those exact strings than to fabricate alternatives. Asirvatham et al. [2.4.45] demonstrated that RAG grounding reduces hallucination rates in knowledge-intensive tasks but does not eliminate them: the LLM may attend to the wrong portion of the prompt, prioritize pretraining knowledge over the injected context, or select a nearby but incorrect value from the retrieved data. RAG grounding reduces the probability of error but provides no mechanism to detect which specific tool calls remain incorrect — the system cannot distinguish a grounded correct call from an ungrounded hallucination.

Human-in-the-loop approval [2.4.46] takes the opposite approach to the same problem: rather than attempting to prevent or reduce errors during generation, it requires a human operator to review each tool call before execution and either approve or reject it. This eliminates hallucination risk entirely — every argument is verified against a human's knowledge of the domain, and no incorrect call reaches an external system. The approach has been deployed in high-stakes decision-support systems where the cost of an error justifies the cost of human review [2.4.46]. The limitation is not accuracy but autonomy: the system requires human staffing for every action, which constrains its applicability to domains where full automation is the objective.

All three approaches share a structural property that limits their effectiveness for autonomous systems. Constrained decoding and RAG grounding operate during generation — they constrain what the LLM can output or influence what it is likely to output — but neither inspects the actual output against an authoritative source after the generation step completes. Human-in-the-loop moves the check after generation but removes autonomy. What none of the three provides is an autonomous post-generation inspection: after the LLM produces a tool call, a deterministic component verifies each argument against a domain-specific source of truth, blocks any call that fails verification, and provides feedback enabling the LLM to produce a corrected call within a bounded number of attempts. Figure 2.10 illustrates this distinction.

**[Figure 2.10 — Generation-time vs. post-generation validation: constrained decoding and RAG grounding operate before or during generation; human review operates post-generation but removes autonomy; a post-generation validator would operate after generation, autonomously, against an authoritative source.]**

| Approach | Syntax Check | Semantic Check | Autonomous | Operates At |
|----------|:---:|:---:|:---:|---|

Each documented approach addresses one dimension of the validation problem at the expense of another. Constrained decoding prevents malformed calls but cannot detect semantic errors — it is syntax-only. RAG grounding reduces the rate of semantic errors probabilistically but provides no detection mechanism for the errors that remain, and the residual error rate after grounding has not been characterized for domain-specific tool calling in Vietnamese. Human-in-the-loop catches both syntax and semantics but eliminates the autonomy that makes an AI agent valuable.

Arranged by the point at which each intervenes, the three approaches leave one position in the pipeline unoccupied. Constrained decoding and RAG grounding act before or during generation, shaping what the model is able or likely to emit. Human review acts after generation, but supplies its judgement from outside the system. The interval between the moment a tool call exists and the moment it executes — the only point at which a proposed action is both fully formed and not yet consequential — is occupied by nothing autonomous in any of the surveyed work. Whether it can be occupied autonomously is a question distinct from the one the three approaches answer. They address how to make a model produce wrong arguments less often; the unaddressed question is whether wrong arguments can be recognised as wrong once produced, by something other than a person, and what a system should do with that recognition.

### 2.4.6 Memory and State Management in Conversational Agents

A restaurant conversation spans multiple turns — a customer browses, asks questions, adds items, removes items, changes their mind, confirms, and pays across five to fifteen exchanges. After turn three, when the customer asks about a dish mentioned in turn two, the agent must resolve the reference. After eight turns of building a cart, the cart must contain every item added and none removed. Four strategies for maintaining dialogue history in an agent's context have been studied, each trading recall fidelity for token efficiency.

The sliding window [2.4.47] retains the most recent N messages and discards everything older. It is simple, bounded in cost, and predictable, but in a fifteen-turn conversation, a window of ten messages silently discards the first five turns, including earlier references. Periodic summarization, as implemented in MemGPT [2.4.48], compresses the conversation every K turns into an LLM-generated summary that replaces raw message history in the context window, reducing token consumption at the cost of a compression step that can introduce errors. Vector-based retrieval, introduced by MemoryBank [2.4.49], embeds each turn and retrieves the most semantically similar prior turns at each new utterance, enabling targeted recall of specific exchanges without processing the full history. Hybrid approaches, as in LongMem [2.4.50], combine summarization with retrieval: a summary provides broad conversational context while retrieved turns provide specific detail from distant history.

All strategies operate within a fundamental constraint: the LLM's context window. The "lost in the middle" phenomenon [2.4.28] and attention boundary preference [2.4.29] mean the model attends poorly to content in the middle regardless of allocated tokens. For Vietnamese the constraint tightens further, since the tokenization penalty described in §2.4.3 applies to every stored turn: the same dialogue occupies more of the window than an English equivalent would, and the budget must additionally accommodate system prompts, few-shot examples, retrieved domain context, and tool-call history alongside the conversation itself.

A distinction that general-purpose memory frameworks do not natively provide is the separation between conversation history — the text of what was said — and application state — what was decided. The cart (item names, quantities, prices) is a structured data object, not conversation text. The order stage (idle, building, confirming, confirmed) is a discrete state variable that gates which tools are valid. Search results from prior turns must persist to prevent redundant retrieval cycles. Dialogue state tracking [2.4.51] established the pattern of per-session state with explicit boundaries, and LangGraph's SQLite-backed checkpointer [2.4.13] implements this by serializing the full agent state — both conversation history and application state fields — after each node execution, keyed by session identifier.

Below: application state (cart, stage, search context) maintained as structured fields alongside conversation text.]**

General-purpose memory frameworks provide mechanisms — sliding windows, summarization, vector retrieval, checkpointing — but do not prescribe how they should be combined for a domain where memory includes both unstructured conversation history and persistent application state with strict correctness requirements. No prior work has characterized a memory architecture that jointly addresses: conversation history retention within the tighter token budget of Vietnamese text, persistent application state maintained as structured fields rather than reconstructed from conversation summaries, strict session isolation between concurrent conversations, and context window allocation across system prompts, few-shot examples, conversation history, retrieved menu context, and tool-call history.

### 2.4.7 Tool Composition, Domain Adaptation, and the Cross-Domain Validation Pattern

Beyond the individual layers surveyed in the preceding subsections, three cross-cutting concerns determine how an agent operates in practice: how it composes multiple tool calls into a valid sequence, how it is adapted to a specific domain without retraining model weights, and what architectural patterns recur across successful domain-specific agent deployments.

The foundational pattern for task-oriented agents is a single LLM selecting and calling tools based on user input. ReAct [2.4.9] established sequential tool execution where each call consumes the output of the previous one. Toolformer [2.4.52] demonstrated that LLMs learn to insert appropriate tool calls through self-supervised training on API documentation. Gorilla [2.4.53] showed accurate tool selection from large API collections through fine-tuning on synthetic tool-call data. The convergence across these systems is that tool selection accuracy has been well-characterized; the less-studied dimension is compositional correctness — whether the sequence of tool calls produces the intended final state regardless of whether each individual call was correct.

When a single utterance demands multiple tools, the composition pattern determines sequence validity. Sequential composition executes tools one after another, with each consuming the previous output. It is simple to implement and debug, but cannot express conditional branching where the choice of the second tool depends on the result of the first. Parallel composition executes independent tools concurrently, relevant when an utterance contains multiple unrelated intents. Conditional composition routes to a second tool based on deterministic evaluation of the first tool's output — after a search returns results, the system must determine whether the top match is sufficiently exact to proceed to ordering, ambiguous enough to require clarification, or empty. Prior benchmarks for tool-using agents have measured per-call accuracy — how often the correct tool is selected — rather than compositional correctness — whether the resulting multi-call sequence is valid and produces the correct state [2.4.54]. Under per-call metrics, a system that calls `confirm_order` before `add_cart` scores correctly on both calls despite producing an empty confirmed order. This measurement gap means that documented tool-using agent accuracy may overstate real-world reliability for transactional domains.

Prompt engineering allows a general-purpose LLM to be adapted to a specific domain without modifying model weights — the only feasible approach when no domain-specific training dataset exists. Three techniques comprise the standard adaptation toolkit. System prompts define the agent's role, behavioral constraints, and output format in a fixed preamble positioned at the context window's beginning, where transformer attention is naturally strongest [2.4.55]. Few-shot examples demonstrate expected behavior through static input-output pairs that include both the reasoning trace and the tool call JSON, showing the model what to call and why [2.4.56]. Dynamic context injection inserts runtime state — conversation history, current application state, prior search results — into the prompt immediately before each LLM call, making the agent situationally aware without encoding session-specific data into fixed prompts. DSPy [2.4.57] formalizes this as an optimization pipeline where prompts and few-shot examples become tunable parameters optimized against an evaluation metric, shifting adaptation from manual tuning to systematic search. No Vietnamese restaurant ordering dataset exists for any of these techniques, meaning that while the methods are documented, their effectiveness on Vietnamese domain-specific tool calling — where dish names, tones, and politeness registers differ from English — has not been assessed.

Across safety-critical domains, successful task-oriented agents share a recurring architectural pattern that transcends individual implementation choices. Healthcare dialogue agents [2.4.58] validate medical recommendations against drug interaction databases and dosage limits before a recommendation reaches a patient. Customer service agents [2.4.59] maintain persistent application state across turns and enforce return policies through deterministic business rules that the LLM cannot override. Code generation agents [2.4.60] validate proposed edits against test suites before committing changes, operating in an edit-validate loop where the LLM proposes and the environment validates. The common pattern — a probabilistic component proposes, a deterministic component validates against authoritative constraints, and only validated proposals become actions — has emerged independently in each of these domains. While the individual instances of this pattern have been documented, the pattern itself has not been formally characterized as a general architectural requirement for systems where LLM output affects real-world state with material consequences.

The literature on conversational agents has produced capable components at every layer — architectures that govern LLM-tool interaction, models that handle Vietnamese text, routers that classify intent, and techniques for grounding generation in authoritative data. What the literature has not produced is an agent where these components are integrated such that the structural constraints at each layer compensate for the probabilistic nature of the LLM at the reasoning layer, yielding a system whose overall reliability exceeds that of any individual component.

Three integration challenges emerge from the preceding survey. First, the layers are interdependent in ways that prior evaluations do not capture. A graph-based architecture (§2.4.2) can structurally mandate a validator node on every path between an LLM call and a tool execution, but the validator's logic — resolving dish names against a menu database, checking state transitions, enforcing quantity bounds — is domain-specific and independent of the orchestration layer (§2.4.5). A state-augmented router (§2.4.4) can resolve context-dependent utterances, but only if the memory layer (§2.4.6) correctly maintains the application state that the router reads. A prompt-only domain adaptation strategy (§2.4.7) can shape the LLM's behavior without fine-tuning, but only if the reasoning engine (§2.4.3) has sufficient Vietnamese capability and context window capacity to process the system prompts, few-shot examples, and dynamic context alongside the conversation. Each layer's effectiveness depends on the layer beneath it, yet no prior work has evaluated these dependencies in combination.

Second, Vietnamese introduces compounding challenges that appear at every layer simultaneously. Teencode abbreviations and informal variants stress the routing layer (§2.4.4) and the reasoning layer (§2.4.3) together — the router must classify informal utterances, and the LLM must produce tool calls using formal dish names that may differ substantially from what the customer said. Tonal diacritics and compound words stress the memory layer (§2.4.6) by consuming more tokens than English equivalents, reducing the effective context window budget that the reasoning layer (§2.4.3) can use for system prompts, few-shot examples, conversation history, and retrieved menu context. The architecture layer (§2.4.2) must accommodate this tighter token budget while still enforcing process, validating actions, and bounding execution. No prior work has characterized how Vietnamese linguistic properties compound the design constraints that each layer imposes on the others.

Third, the validation-gated execution pattern that has emerged independently across healthcare, customer service, and code generation (§2.4.7) has not been combined with the other five layers for a Vietnamese task-oriented dialogue agent. A graph-based architecture can provide the structural governance, a state-augmented router can provide deterministic intent classification, a post-generation validator can inspect tool call arguments against an authoritative source, a session-scoped memory can maintain application state across turns, and prompt-only adaptation can shape the LLM's behavior without fine-tuning — but no prior system has integrated all of these into a single agent evaluated on Vietnamese restaurant conversations. The integration of these layers, where each layer's structural constraints address a specific failure mode of the probabilistic reasoning engine, represents the primary gap that the survey identifies.
