## 2.8 Edge Computing Platform

> *The robot's computational platform is a purchased component — no custom hardware was developed. This section describes the NVIDIA Jetson Orin Nano: its hardware specifications, the software stack that makes GPU-accelerated inference available on an embedded device, and the sensors that interface with it. The platform's resource constraints motivate the architectural decisions in Chapter 4.*
>
> **Cross-refs:** §2.2 (navigation sensors), §2.3 (voice pipeline hardware requirements), §3.3 (robot hardware setup — sensor wiring), §4.4.2 (edge/server workload split)

The NVIDIA Jetson Orin Nano Developer Kit is a single-board computer purpose-built for embedded AI and robotics workloads. It is sold as an off-the-shelf product with a pre-installed operating system and a complete GPU-accelerated software stack — no custom hardware design was required for this project beyond mounting the board on the robot chassis and connecting the sensors.

Physically, the board measures approximately 100 × 79 mm, small enough to mount inside the robot's electronics bay alongside the motor controller and battery pack. It is passively cooled via a heatsink and optional fan, with a power envelope of 7 to 15 watts depending on the configured performance mode. At 10 watts — the default mode for sustained operation under load — the board draws manageable current from the robot's 12V battery through a step-down regulator, leaving sufficient power budget for the motors, LiDAR, and RealSense camera.

The Orin Nano's compute resources are defined by three specifications that together determine what workloads the robot can run concurrently. First, its GPU is a 1024-core NVIDIA Ampere architecture with 32 Tensor Cores — the specialized compute units that accelerate neural network inference. The Tensor Cores execute mixed-precision matrix operations in hardware, the same mechanism that powers GPU inference on data center cards, but at a fraction of the scale. This GPU is capable of running transformer-based speech recognition models at real-time speed, as the benchmarks for faster-whisper on Jetson hardware have demonstrated [2.8.1]. It is not, however, capable of running a 7-billion-parameter language model at acceptable latency within the available memory budget — a constraint whose implications are explored in Chapter 4.

Second, its CPU is a 6-core ARM Cortex-A78AE processor, an architecture designed for embedded and automotive applications where deterministic real-time behavior matters as much as throughput. The ARM cores run the Ubuntu operating system, the ROS2 middleware, the sensor drivers, and the voice pipeline's control logic (VAD frame processing, Piper TTS inference). They do not directly run GPU-accelerated workloads, but they orchestrate them by loading model weights onto the GPU and dispatching inference requests through NVIDIA's CUDA runtime.

Third, and most architecturally significant, is the memory architecture. The Orin Nano uses 8 GB of LPDDR5 unified memory — a single physical pool shared between the CPU and GPU. There is no separate CPU RAM and GPU VRAM; every process running on the board, whether it is the Linux kernel, the ROS2 navigation stack, a LiDAR driver, the speech recognition model, or the text-to-speech voice, draws from the same 8 GB allocation. When the combined memory usage of all running processes exceeds 8 GB, the kernel's Out-Of-Memory killer terminates the process with the largest allocation — which on a robot, where motor control and obstacle avoidance are live, is a catastrophic failure mode. The unified memory architecture means that the question of what runs on the Jetson is exclusively a question of what fits within 8 GB concurrently.

The Orin Nano occupies a specific position within the Jetson product family. Above it sits the AGX Orin with 32 GB of unified memory — sufficient to run an LLM alongside navigation and voice, but at a retail price of approximately 1,999 USD, a cost disproportionate to a platform that does not need to run the LLM locally when a server with a dedicated GPU is already available on the restaurant LAN. Below it sits the older Jetson Xavier NX, widely used in prior academic ROS2 robotics projects, but now discontinued and superseded by the Orin family. The Orin Nano sits at the intersection of sufficient GPU capability for real-time speech inference, sufficient CPU headroom for ROS2 processes, and a cost envelope appropriate to an academic robotics project. This positioning, and the memory budget analysis presented in Chapter 4, determine which workloads run where.

### Software Stack

The Jetson platform ships with JetPack SDK, NVIDIA's integrated software distribution that bundles an operating system, GPU drivers, and accelerated libraries into a single installable image [2.8.2]. The operating system is Linux for Tegra (L4T), a customised build of Ubuntu 22.04 LTS compiled for the ARM64 architecture. From a developer's perspective, L4T presents a standard Ubuntu environment: `apt` for package management, Python 3.10 as the system interpreter, Bash as the shell, and the same POSIX APIs that any Linux application expects. ROS2 Humble, which targets Ubuntu 22.04 as its tier-1 platform, installs natively through its standard Debian repositories without cross-compilation or architecture-specific patches.

The GPU acceleration stack consists of three libraries that together form the inference pipeline for the voice models. CUDA 12.6 provides the low-level GPU programming interface — kernel launches, memory allocation on the unified pool, stream management for concurrent execution [2.8.3]. cuDNN provides GPU-accelerated primitives for deep neural networks: convolution, pooling, activation functions, and the attention matrix operations that transformer models depend on. TensorRT is NVIDIA's inference optimization compiler, which applies layer fusion, kernel auto-tuning, and precision calibration to a trained model, producing an optimized execution plan. The CTranslate2 engine that powers faster-whisper leverages these libraries to achieve approximately 4× inference speedup over the standard Whisper implementation on ARM64 GPUs [2.8.4].

This software stack is mature and well-supported. NVIDIA provides regular JetPack releases synchronized with Ubuntu LTS versions. The libraries are binary-compatible across the Jetson family — a model optimised with TensorRT on an Orin Nano runs on any other Orin-series device without recompilation. Prior work has validated this stack extensively for ROS2 navigation workloads (SLAM, sensor fusion, Nav2) and for English-language edge AI inference (object detection, speech recognition). Vietnamese-language edge deployment on Jetson is less documented in the literature, but the underlying inference libraries are language-agnostic — they accelerate matrix operations regardless of which language the model processes.

### Sensor Interfaces

The Jetson board provides the physical connectivity through which all robot sensors and actuators communicate with the software stack. Five sensor systems connect to the Jetson, each through a standard interface [2.8.5]:

The RPLiDAR A2M8 connects via USB 2.0 and streams 360-degree 2D laser scans at 8 Hz through a serial-over-USB protocol. The ROS2 `rplidar_ros` driver reads the scan data and publishes `sensor_msgs/LaserScan` messages to the ROS2 graph, where they are consumed by the SLAM and Nav2 costmap layers. The Intel RealSense D435 connects via USB 3.0 and streams 1280 × 720 RGB frames and depth frames at 30 Hz. The higher bandwidth of USB 3.0 is necessary because the depth frames alone consume approximately 55 MB/s — a data rate that USB 2.0 cannot sustain. The ROS2 `realsense2_camera` driver publishes colour images, depth images, and point clouds; the ArUco marker detection node subscribes to the colour stream for docking, while RTAB-Map subscribes to the depth stream for visual loop closure detection.

The MPU6050 inertial measurement unit does not connect directly to the Jetson. It is soldered onto the STM32 motor controller board and communicates with the STM32 over I²C. The STM32 reads the accelerometer and gyroscope registers at 200 Hz, packages the raw values with the encoder tick counts from the motor encoders, and transmits the combined packet to the Jetson over a UART serial link. A custom ROS2 node running on the Jetson parses this packet, converts the raw integer values to SI units, and publishes `sensor_msgs/Imu` messages. The `robot_localization` EKF node subscribes to these IMU messages and fuses them with wheel odometry — the sensor fusion approach described in §3.4.

The USB conference microphone connects via USB 2.0 and appears to the operating system as a standard USB Audio Class device. PyAudio opens the device at 16 kHz mono 16-bit PCM — the sample rate expected by the Silero VAD and faster-whisper models. A Bluetooth speaker connects over the Bluetooth A2DP profile for TTS audio output; the alternative is a wired speaker through the Jetson's 3.5 mm audio jack, which avoids the pairing and latency variability of Bluetooth at the cost of a physical cable routed through the robot chassis.

The 7-inch LCD touchscreen connects via HDMI for video and USB for touch input. It serves as the robot's local display, running the customer tablet interface directly on the robot in scenarios where a wall-mounted tablet is not available.

These sensor interfaces are standard and well-documented. The contribution is not in the interfaces themselves but in their concurrent operation — LiDAR scans at 8 Hz, camera frames at 30 Hz, IMU readings at 200 Hz, microphone capture at 16 kHz, and TTS playback, all on a single USB and I/O controller — which imposes bandwidth and driver scheduling constraints that the ROS2 node graph and the threaded voice pipeline must accommodate. How the system handles this concurrency is described in §4.4.

### Prior Work on Jetson in Robotics

The Jetson platform has been deployed extensively in academic and research robotics projects [2.8.6]. In ROS2-based mobile robots, it serves as the primary compute platform for simultaneous localization and mapping (Cartographer, RTAB-Map), autonomous navigation (Nav2 with global and local planners), sensor fusion (robot_localization EKF), and computer vision (YOLO-based object detection, ArUco marker recognition). These use cases are individually validated and well-represented in the literature, with documented performance benchmarks on Jetson hardware across multiple ROS2 distributions.

The combination that has not been extensively characterised — and which drives the architectural decisions in this project — is the concurrent execution of the ROS2 navigation stack and a Vietnamese voice processing pipeline on the same Jetson device. Prior work evaluates these workloads in isolation: a robot that navigates, or a voice assistant that transcribes speech. Running both simultaneously, where the GPU must context-switch between SLAM localisation, speech transcription, and text-to-speech inference, introduces resource contention whose behaviour on a 1024-core Ampere GPU with 8 GB of unified memory is not documented in the existing robotics literature. This concurrent-workload scenario, and the memory budget analysis it necessitates, is addressed in Chapter 4.
