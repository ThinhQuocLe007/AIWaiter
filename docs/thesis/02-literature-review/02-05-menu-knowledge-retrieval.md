## 2.5 Menu Knowledge Retrieval

> *Retrieval-augmented generation (RAG) grounds LLM outputs in domain-specific documents that were not present during training. This section surveys the RAG pipeline bottom-up: why retrieval is necessary and how the standard architecture evolved from naive embedding-based search to advanced and modular designs (§2.5.1); the embedding models — dense bi-encoders and sparse keyword models — that convert text into searchable representations, along with the Vietnamese word segmentation prerequisite for both paradigms (§2.5.2); how these representations are stored in searchable indices — dense vector indices and sparse inverted indices — and queried at runtime (§2.5.3); how results from multiple retrieval strategies are fused into a single ranking (§2.5.4); and the pipeline extensions proposed beyond the standard retrieve-then-generate paradigm — query rewriting, post-retrieval evaluation, and multi-turn search context — each evaluated in isolation but never composed into an architecture where the LLM actively controls retrieval quality (§2.5.5).*
>
> **Cross-refs:** §2.1 (overview — RAG limitations), §4.1 (FR2 — menu search by attributes), §4.6 (knowledge retrieval pipeline — proposed method), §5.3.4 (retrieval evaluation)
> **Citations:** [2.5.1]–[2.5.40]; final numbering assigned when all Ch.2 references are merged.

### 2.5.1 The Knowledge Problem and Standard RAG

Large language models are trained on public web data frozen at a specific point in time. Domain-specific knowledge — a restaurant's menu, with its particular dish names, local ingredients, seasonal items, and precise pricing — is absent from the training distribution. When queried about such knowledge without access to it, LLMs produce factually incorrect outputs with high confidence, a failure mode documented as closed-book hallucination [2.5.1]. Retrieval-augmented generation (RAG), introduced by Lewis et al. [2.5.2], decouples knowledge from model parameters by introducing a retrieval step between the user's query and the LLM's generation. Offline, a knowledge corpus is split into document chunks, each embedded into a dense vector representation using a sentence encoder, and stored in a vector index. Online, the user's query is embedded with the same encoder, the top-k documents with the most similar vectors are retrieved via similarity search, and those documents are injected into the LLM's prompt as grounding context. This architecture — embedding, retrieval, generation — has become the dominant approach for knowledge-intensive NLP tasks including open-domain question answering [2.5.3], fact verification [2.5.4], and domain-specific dialogue [2.5.5].

The standard RAG pipeline, now termed naive RAG in the survey literature [2.5.6], has been extended along two trajectories. Advanced RAG improves individual stages of the naive pipeline: chunk optimization strategies such as semantic splitting and sliding-window overlap produce document segments with coherent semantic boundaries rather than arbitrary length-based cuts; re-ranking models trained on retrieval relevance scores reorder the top-k retrieved documents to improve precision at the cost of an additional inference step; query expansion augments the user's query with synonyms, related terms, or sub-questions before embedding, broadening the retrieval net. These improvements are incremental — each addresses a specific failure mode (poor chunk boundaries, imprecise ranking, narrow queries) without changing the fundamental architecture. The LLM remains a downstream consumer of whatever retrieval returns.

Modular RAG [2.5.7] restructures the pipeline into independent, swappable modules — a retriever module, a rewriter module, an evaluator module, a generator module — with configurable composition. This modularity enables per-domain optimization: a medical QA system might prioritize retrieval precision through aggressive re-ranking; a creative writing assistant might configure the retriever for recall over precision. The modular architecture exposes a design question that naive and advanced RAG leave implicit: which modules should execute, in what order, and under what conditions? The literature documents the configurable composition [2.5.7] but does not address the coordination problem — how a module should decide whether its output is adequate, whether to invoke another module, or whether to signal that retrieval has failed. The modules exist independently; the control logic connecting them is absent from the documented architecture.

All three RAG variants — naive, advanced, and modular — share a single structural assumption: the query embedding lies close to the relevant document embeddings in vector space. This assumption holds when the query and the target documents share vocabulary. It fails when the query describes information needs through terms absent from the document corpus, a failure mode documented as query-document vocabulary mismatch [2.5.8]. The mismatch is structural — not a matter of index quality, encoder choice, or chunk strategy — because the query vectors and the document vectors occupy disconnected regions of the embedding space. The retrieval failure cannot be fixed by improving any individual pipeline stage; it requires an architectural position that does not exist in naive, advanced, or modular RAG: a component that inspects the query, recognizes the vocabulary gap, and transforms the query into terms the document space can match before retrieval executes.

— Gap. The standard RAG pipeline, across all three documented variants, assigns the LLM a passive role: retrieval happens, then the LLM generates from the retrieved documents. The LLM does not decide which retrieval strategy to use for a given query, does not inspect whether the retrieved results are satisfactory, and has no mechanism to adjust its approach when retrieval fails. No prior RAG architecture provides an active control role for the LLM — a position where the model reasons about retrieval quality, decides on a strategy, and takes corrective action when retrieval produces noise or emptiness. The extensions surveyed in §2.5.5 — query rewriting, post-retrieval evaluation, multi-turn context — address pieces of this gap individually but have never been composed into a single control loop. This architectural gap is the foundation for the remaining subsections.

### 2.5.2 Embedding Models

> *Embedding models convert text into representations that can be compared for relevance. Two paradigms have been studied: dense embeddings produce continuous vectors in a learned semantic space; sparse embeddings produce high-dimensional term-weight vectors representing exact lexical content. Vietnamese word segmentation is a shared prerequisite: without correct word-boundary identification, both paradigms represent syllable fragments rather than lexical units. This section surveys all three — segmentation tools, dense models, and sparse models — as off-the-shelf components whose documented properties enable informed selection.*

Vietnamese script places spaces between syllables, not between words. A three-syllable compound such as "bún bò Huế" is written as three space-separated tokens but constitutes one lexical item — a specific spicy beef noodle soup. Without word segmentation, both dense and sparse models represent per-syllable fragments: the embedding for "bún" + "bò" + "Huế" separately rather than the compound as a whole, or the BM25 index treats "bún," "bò," and "Huế" as independent terms, matching any document containing any of the three syllables. Three tools have been documented for Vietnamese word segmentation [2.5.9]. `underthesea` applies a CRF-based segmentation model trained on the Vietnamese Treebank, outputting compound tokens joined by underscores; it achieves approximately 97% accuracy on the VLSP 2013 word segmentation benchmark for formal text, runs as a pure Python library with approximately 50 MB of model data, and is the most widely integrated option in the Vietnamese NLP ecosystem. `VnCoreNLP` is a Java-based pipeline including segmentation, part-of-speech tagging, named entity recognition, and dependency parsing; its richer feature set — word embeddings, character-level RNN representations — yields higher documented accuracy than `underthesea` on formal Vietnamese text but introduces a Java runtime dependency that adds deployment complexity and per-call process overhead. `pyvi` is a lightweight pure-Python tokenizer using dictionary-based segmentation with regular-expression post-processing; its accuracy is lower than the alternatives, with documented weakness on compound terms where syllable boundaries are ambiguous — a limitation directly relevant to a retrieval domain dominated by compound dish names. No prior work has benchmarked segmentation accuracy specifically on domain-specific proper nouns, where the segmenter's dictionary and statistical model are least reliable.

| Tool | Method | Accuracy (VLSP 2013) | Deployment | Documented Limitations |
|------|--------|:---:|---|---|
| `underthesea` | CRF, Vietnamese Treebank | ~97% | Pure Python, ~50 MB | Accuracy on informal/elided forms unbenchmarked; domain-specific proper nouns not evaluated |
| `VnCoreNLP` | RNN + word embeddings + CRF | Higher (~98%+) | Java (separate process) | Java runtime dependency; per-call process overhead vs. in-process alternatives |
| `pyvi` | Dictionary + regex | Lower | Pure Python, no model | Weak on compound terms with ambiguous boundaries; dictionary coverage limited |

Dense embedding models produce continuous vector representations through sentence encoders trained on paired or contrastive objectives. Two categories of models have been documented for Vietnamese text.

Vietnamese-native bi-encoders are trained specifically or predominantly on Vietnamese corpora. `bkai-foundation-models/vietnamese-bi-encoder` [2.5.10] uses a PhoBERT-base encoder [2.5.11] pre-trained on a 20 GB Vietnamese corpus and fine-tuned on Vietnamese sentence similarity pairs with L2 normalization. Its monolingual Vietnamese tokenizer preserves diacritics without the subword boundary artifacts that multilingual BPE tokenizers introduce; its training corpus includes informal registers from social media and forums, providing stronger representation of casual orthographic variants than models trained exclusively on formal text. `VoVanPhuc/sup-SimCSE-VietNamese-phobert-base` [2.5.12] applies the SimCSE contrastive learning framework [2.5.13] to PhoBERT, evaluated on Vietnamese semantic textual similarity benchmarks. `AITeamVN/Vietnamese_Embedding` [2.5.14] uses a 1,024-dimensional space from a BART-based encoder, trading larger index size and longer search latency for higher representational capacity. Vietnamese-native models have been evaluated on Vietnamese STS and ViText2Vec benchmarks [2.5.15] where they outperform multilingual alternatives by 8–15% on cosine similarity correlation, but retrieval-specific benchmarks — measuring precision and recall on document collections rather than sentence-pair similarity — are unreported for this model category.

Multilingual models with Vietnamese capability achieve state-of-the-art results on cross-lingual retrieval benchmarks that include Vietnamese queries. `BAAI/bge-m3` [2.5.16] is a 568M-parameter model supporting over 100 languages; it provides native dense, sparse, and multi-vector retrieval through a unified architecture and is currently the top-ranked model on the MIRACL multilingual retrieval benchmark. `intfloat/multilingual-e5-large` [2.5.17] applies the E5 contrastive training recipe to a multilingual encoder, achieving competitive performance on cross-lingual retrieval tasks. `sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2` [2.5.18] serves as a compact baseline at 384 dimensions and 118M parameters, widely integrated in RAG frameworks but with documented weaknesses: its BPE tokenizer strips Vietnamese diacritics during subword tokenization, collapsing tonal distinctions into embedding noise, and its reduced dimensionality sacrifices representational resolution relative to 768-dim alternatives. Multilingual models handle Vietnamese diacritics partially — the tokenizer preserves diacritic-bearing characters but does not allocate attention specifically for tonal distinctions, and compound-word boundaries are not recognized as syntactic units distinct from syllable spans.

The selection among these models involves a documented trade-off between Vietnamese-language fidelity and ecosystem maturity. Vietnamese-native models provide stronger diacritic handling and are trained on Vietnamese-register text, but their retrieval performance is less documented and they lack features that multilingual alternatives provide natively (multi-vector retrieval, cross-lingual search). Multilingual models offer mature retrieval features and broad benchmark coverage but with Vietnamese diacritic handling that is adequate rather than optimized. The benchmark gap is significant: the cross-lingual retrieval benchmarks (MIRACL, MTEB) that evaluate multilingual models use documents — news articles, encyclopedia entries — whose vocabulary distribution, document length (200–500+ tokens), and term-frequency patterns differ substantially from domain-specific corpora composed of short structured documents (30–50 tokens) dominated by rare proper nouns.

|-------|-----|:---:|:---:|:---:|---|---|
| `bkai-foundation-models/vietnamese-bi-encoder` | 768 | ✓ | ✓ | ~135M | Trained on Vietnamese sentence pairs including informal registers; L2-normalized; teencode-aware | Single-language only; code-switching not supported; retrieval-specific benchmarks unreported |
| `VoVanPhuc/sup-SimCSE-VietNamese-phobert-base` | 768 | ✓ | ✓ | ~135M | SimCSE contrastive training on PhoBERT; evaluated on Vietnamese STS | Encoder-only; less evaluated on retrieval tasks than on similarity tasks |
| `AITeamVN/Vietnamese_Embedding` | 1024 | ✓ | ✓ | Not specified | Larger dimension; BART-based encoder with contrastive fine-tuning | Larger index size and search time; benchmark coverage narrower than alternatives |

| `intfloat/multilingual-e5-large` | 1024 | ✗ (multilingual) | Partial | 560M | Top-ranked on cross-lingual retrieval; E5 contrastive recipe | Primarily high-resource language evaluation; Vietnamese domain-specific effects uncharacterized |
| `sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2` | 384 | ✗ (multilingual) | Weak | 118M | Compact, fast inference, widely integrated | 384-dim reduced representational resolution; BPE tokenizer strips diacritics in Vietnamese |

Sparse keyword models represent the alternative paradigm to dense embeddings. Rather than learning a continuous semantic space, they represent documents as sparse vectors indexed by vocabulary terms, with weights determined by term frequency and inverse document frequency statistics. BM25, introduced by Robertson and Walker [2.5.19], extends the TF-IDF framework with document-length normalization through two tunable parameters: k₁ controls term-frequency saturation (preventing a term that appears 50 times from being treated as 50× more relevant than one that appears 5 times), and b controls the degree of length normalization (a value of 0 disables length normalization entirely, appropriate when all documents are similarly short). BM25's primary strength is exact keyword matching: a document whose indexed terms precisely match query terms receives high scores with strong discriminability between documents containing the terms and those that do not. Its primary limitation is the vocabulary gap: a query containing no terms present in the document index produces zero results, regardless of parameter tuning or term weighting, because the sparse representation has no mechanism to bridge a lexical gap.

For Vietnamese, BM25 effectiveness depends on word segmentation quality. If compound terms are split into independent syllable tokens, BM25 matches individual syllables that appear across unrelated documents — a query syllable "bò" matches both "bún bò Huế" (correct, contains beef) and "bò lúc lắc" (wrong, a completely different dish that happens to contain beef) — producing false positives. If segmentation is too conservative, compound terms fail to match partial-name queries where the user provides only the head noun. The interaction between segmentation quality and BM25 retrieval precision has not been systematically studied for Vietnamese domain-specific corpora.

| Parameter | Role | Vietnamese-Specific Consideration |
|-----------|------|----------------------------------|
| k₁ (term frequency saturation) | Prevents a term appearing 50 times from scoring 50× a term appearing 5 times. Typical range: 1.2–1.5 | No language-specific effect; standard IR parameter |
| b (length normalization) | Controls document-length penalty. b = 0 disables normalization entirely — appropriate when all documents are similarly short | Enables fixed-length normalization for menu entries (all documents 30–50 tokens) without penalizing slightly longer compound dish names |
| Tokenization dependency | BM25 vocabulary is the set of tokens produced by the word segmenter. Mis-segmented compounds fragment into independent terms | Compound dish names must be preserved as single tokens; without correct segmentation, individual syllables match across unrelated documents |
| Strengths | Exact keyword matching with high discriminability between documents containing and not containing query terms | Effective for dish-name lookup when segmentation is correct |
| Limitations | Zero vocabulary overlap produces zero results regardless of parameter tuning | Sensory queries with no menu vocabulary return empty results; no mechanism to bridge the lexical gap |

### 2.5.3 Indexing and Search

> *Once text is converted to representations — dense vectors from a bi-encoder, or sparse term-weight vectors from BM25 — those representations must be stored in a searchable index and queried efficiently. The indexing strategy follows from the representation type: dense vectors require a vector index supporting similarity search over continuous spaces; sparse vectors require an inverted index mapping vocabulary terms to document postings. Both indexing paradigms are mature, off-the-shelf technologies with well-documented performance characteristics.*

Dense vector indexing is dominated by FAISS (Facebook AI Similarity Search), introduced by Johnson et al. [2.5.20]. FAISS provides a family of index structures spanning the accuracy-speed trade-off. At one end, `IndexFlatL2` performs exhaustive comparison of the query vector against every stored vector, computing exact Euclidean distances; it is deterministic — the same query always returns the same results — and sufficient for small corpora of a few hundred to a few thousand documents, where the O(n) per-query cost is negligible. At the other end, approximate indices such as `IndexIVFFlat` partition the vector space into Voronoi cells via k-means clustering, searching only the cell closest to the query vector and its neighbors, reducing search complexity to sub-linear at the cost of a small probability of missing the true nearest neighbor. `IndexHNSW` builds a navigable small-world graph over the vectors, enabling logarithmic-time approximate search through greedy graph traversal. The distance metric — cosine similarity via L2 distance on normalized vectors, inner product for unnormalized representations, or Euclidean distance — determines the definition of "similar" in vector space. FAISS stores both the vector and a back-reference to the original document; retrieval returns the document and a similarity score.

Sparse inverted indexing maps each vocabulary term to a postings list — the set of documents containing that term, each annotated with term frequency, positions within the document, and any payload data. At query time, the query is tokenized into terms, the postings list for each term is retrieved, and BM25 scoring is computed over the intersection of query terms and document postings. The inverted index is the foundational data structure of information retrieval, validated across decades of evaluation at TREC, CLEF, and NTCIR [2.5.21]. Its performance characteristics are well-understood: index construction is O(n) in the number of documents, query time is O(|Q| × avg_posting_length) where |Q| is the number of query terms, and memory footprint is proportional to the vocabulary size and average document length. For Vietnamese, the vocabulary depends on word segmentation quality: mis-segmented compound terms fragment into multiple postings lists, inflating index size and diluting term specificity.

| Property | Dense (FAISS) | Sparse (Inverted Index) |
|----------|:---:|:---:|
| Search mechanism | Similarity search in continuous vector space: cosine, inner product, or Euclidean distance | Term intersection over postings lists with BM25 scoring |
| Index types | Flat (exact, O(n)), IVF (approximate, sub-linear via clustering), HNSW (approximate, logarithmic via graph traversal) | Postings lists with per-document term frequency and position data |
| Determinism | Flat search is deterministic; approximate indices trade deterministic correctness for speed | Deterministic for a given tokenization |
| Corpus scale | Flat sufficient for small corpora (hundreds of documents); approximate indices for large scale | Efficient for any scale; construction O(n), query O(|Q| × avg_posting_length) |
| Vocabulary-gap handling | Encoder-dependent — the embedding model determines whether a query with no lexical overlap can still retrieve relevant documents | No handling — zero vocabulary overlap produces zero results |
| Vietnamese dependency | Embedding model choice determines diacritic sensitivity and compound-word representation quality (§2.5.2) | Word segmentation quality determines whether compound terms are single indexed tokens or fragmented syllables (§2.5.2) |

### 2.5.4 Result Fusion

> *When multiple retrieval strategies execute in parallel — a dense FAISS search and a sparse BM25 search, for instance — they produce separate ranked lists that must be combined into a single ranking. Fusion methods address the fundamental incommensurability of scores from different retrievers: BM25 scores are unbounded term-weight sums; cosine similarities are bounded in [−1, 1]; their numeric ranges carry no common meaning. This section surveys three fusion techniques documented in the information retrieval literature, each approaching the incommensurability problem differently. All are off-the-shelf methods whose documented properties enable informed selection.*

Reciprocal Rank Fusion (RRF), introduced by Cormack et al. [2.5.22] for meta-search evaluation, eliminates the score normalization problem by operating on document ranks rather than raw scores. For each document, the fused score is the sum of the reciprocal of its rank in each constituent result list, shifted by a constant k: score(d) = Σ 1 / (k + rankᵣ(d)), where k is typically set to 60. A document ranked first in both retrievers receives 1/61 + 1/61; a document ranked first in one retriever and absent from the other receives 1/61 from the first and nothing from the second. RRF requires no score normalization, no domain-specific parameter tuning, and no assumption about score distributions. The constant k controls the influence of high-ranked versus low-ranked documents: at k = 60, the score difference between rank 1 (1/61 ≈ 0.0164) and rank 5 (1/65 ≈ 0.0154) is approximately 6%, reflecting the design principle that the top several results from each retriever should contribute with comparable weight. Documents appearing in only one result list are not penalized beyond the absence of contribution from the other retriever — they can still rank highly if their single-list rank is competitive.

Linear combination addresses fusion through score normalization followed by weighted summation: score(d) = α × normalize(s₁(d)) + (1 − α) × normalize(s₂(d)). The method requires per-retriever score normalization — typically min-max scaling to [0, 1] or z-score standardization — and a domain-tuned weight α that controls the relative contribution of each retriever. When α is well-calibrated for a specific domain and document collection, linear combination can achieve higher accuracy than RRF because it exploits score magnitude information that RRF discards [2.5.23]. However, the optimal α does not transfer across domains or document collections — a weight tuned on news articles will not be optimal for menu entries — and the method is sensitive to the choice of normalization scheme. Without domain-specific calibration data, linear combination can underperform RRF because score magnitude differences between retrievers that carry no meaningful signal are inadvertently amplified by the weight.

Condorcet voting [2.5.24] approaches fusion as a pairwise comparison problem: for every pair of documents in the union of the result lists, the document ranked higher in more retrieval lists "wins" the pairwise comparison. The fused ranking is constructed by sorting documents by the number of pairwise wins. The method requires no score normalization and can theoretically capture consensus across more than two retrieval lists naturally. Its computational cost is O(n²) in the number of unique documents across all result lists — a cost that is manageable for small n (tens of documents) but prohibitive when result lists are long. For two-retriever fusion specifically, no evaluation has demonstrated a systematic accuracy advantage over the simpler and faster RRF baseline.

| Method | Score Normalization | Domain Transfer | Computational Cost | Single-List Documents |
|--------|:---:|:---:|:---:|---|
| RRF | Not required — operates on ranks | Domain-independent (k = 60 is fixed; no per-domain tuning needed) | O(n log n) for sorting n unique documents | Receive contribution from one retriever; no penalty for absence from other lists |
| Linear combination | Required — min-max or z-score per retriever | Weight α must be re-tuned per domain and document collection | O(n) after normalization | Score depends on normalization of the present retriever; absent retriever contributes 0 |
| Condorcet voting | Not required — uses pairwise rank comparisons | Domain-independent | O(n²) for n unique documents | Win against documents ranked lower in all lists where both appear; absent documents lose all pairings |

### 2.5.5 Beyond Retrieve—Generate: Rewriting, Evaluation, Context

> *The standard RAG pipeline — embed query, retrieve documents, generate — operates in one direction. Retrieval produces a ranked list; the LLM consumes that list and produces text. There is no architectural position for inspecting retrieval quality, for deciding that the retrieved documents are insufficient, or for feeding a quality signal back to trigger a different retrieval strategy. Three classes of extensions have been proposed in the literature to address pieces of this limitation: pre-retrieval query transformation, post-retrieval result evaluation, and multi-turn search context persistence. Each has been evaluated as an independent point solution on English benchmarks. None addresses the interaction between rewriting, retrieval, evaluation, and context within a single pipeline where the LLM actively controls retrieval quality.*

Pre-retrieval query rewriting transforms the user's query before it reaches the retriever. Three techniques have been documented, each with a different mechanism for bridging the vocabulary gap between query and documents.

HyDE (Hypothetical Document Embeddings), introduced by Gao et al. [2.5.25], prompts an LLM to generate a plausible relevant document — the "hypothetical document" — and embeds that document for retrieval instead of the raw query. The insight is that LLM-generated text, even if factually incorrect, shares vocabulary and structure with real documents in the target corpus, whereas the user's natural-language query may use language from a different register. Evaluated on TREC DL and Web Questions, HyDE improved retrieval recall on vocabulary-gap queries in English. Its documented limitation is unbounded generation: the LLM may fabricate terms, entities, or categories not present in the real corpus, and the hypothetical document's embedding pulls retrieval toward the fabrication rather than toward genuinely relevant documents. The technique provides no mechanism for the LLM to inspect whether the rewritten query improved retrieval — the generation is one-way, from LLM to retriever.

Step-Back Prompting, introduced by Zheng et al. [2.5.26] for reasoning tasks and subsequently adapted to retrieval, takes a different approach: rather than generating a hypothetical document, the LLM abstracts the query to a higher conceptual level and retrieves against the abstraction. An implementation might abstract "món gì ấm bụng cho ngày lạnh" to "món nước nóng, lẩu, súp, cháo" — the categories of food that satisfy the sensory description, expressed in terms the retriever can match. The approach reduces HyDE's hallucination risk because the LLM produces keywords and category labels rather than free-form documents, but it depends on the LLM correctly identifying the appropriate abstraction level: too abstract and the retrieval loses specificity; too specific and the vocabulary gap persists. Evaluated on MMLU and TimeQA in English; no evaluation exists for Vietnamese where the LLM's cultural associations — the mapping from sensory experience to culinary category — are less grounded than its English factual knowledge.

Query2Doc [2.5.27] combines both strategies: the LLM generates a rewritten query string and a hypothetical document, concatenating their embeddings for retrieval. The approach leverages the complementary strengths of keyword rewriting (precision from targeted terms) and hypothetical document generation (recall from shared vocabulary structure). Its documented limitation is computational cost — two LLM inference calls per retrieval operation — which creates a latency barrier for real-time deployment.

All three techniques share a structural property that limits their integration into a control loop: they transform the query and then hand off to retrieval, with no return path. The LLM that performed the rewriting never sees the retrieved results, cannot evaluate whether the rewrite helped or hurt, and cannot adjust its strategy based on retrieval quality. The rewriting is an open-loop transformation.

Post-retrieval evaluation inspects the retrieved documents after retrieval and before generation, filtering noise and, in some architectures, triggering corrective action.

Self-RAG, introduced by Asai et al. [2.5.28], uses reflection tokens — special tokens the model learns to generate during fine-tuning that indicate whether each retrieved passage is relevant to the query and whether the generated text is supported by it. The technique embeds quality assessment into the generation process itself, enabling per-passage relevance filtering without a separate evaluation model. Its primary limitation is the fine-tuning requirement: the reflection tokens must be trained into the model's vocabulary and output distribution, which requires training infrastructure and labeled data that are not available for most domain-specific deployments.

CRAG (Corrective RAG), introduced by Yan et al. [2.5.29], performs relevance scoring at inference time without fine-tuning: the LLM evaluates each retrieved document on a numeric relevance scale, discards those below a threshold, and if no documents pass the threshold, triggers a fallback — typically a web search or an alternative retrieval strategy. The approach adds one LLM inference call per retrieval operation. Evaluated on English QA benchmarks including PopQA, PubHealth, and Arc-Challenge; its effectiveness depends on the LLM's ability to correctly score relevance for the target domain and language. The technique performs evaluation but its corrective action — triggering a fallback search — is predetermined (web search) rather than strategy-aware (the LLM does not decide what kind of search to run next or how to modify its approach).

FLARE (Forward-Looking Active Retrieval augmented generation) [2.5.30] interleaves generation and retrieval: as the LLM generates text token by token, it monitors its own confidence, and when it encounters a token it is uncertain about, it pauses generation, retrieves additional information related to the uncertain span, and resumes generation with the new context. The technique is designed for long-form generation tasks where information need emerges progressively during writing. Its computational cost — multiple retrieval calls interleaved with generation steps — makes it unsuitable for latency-constrained settings where each retrieval call adds measurable delay.

All three post-retrieval techniques have been evaluated on English question-answering and fact-verification benchmarks where relevance is a function of factual accuracy. None has been evaluated on domain-specific retrieval where relevance depends on structured metadata matching — whether a dish's ingredients, taste profile, and preparation method align with a sensory description — rather than factual concordance with a knowledge base.

Multi-turn search context addresses a limitation specific to conversational retrieval: within a single dialogue session, the user may issue multiple retrieval queries that are semantically related to prior turns. A user who searches for a term in turn 3 and then refers to it pronominally in turn 6 ("cái đó có cay không?") expects the system to resolve the reference against prior search results rather than executing a new retrieval on the literal query string "cái đó có cay." Dialogue state tracking, a well-established approach for task-oriented dialogue systems [2.5.31], maintains structured per-session state — slot values, dialogue acts, user goals — but is designed for finite-state or frame-based architectures and is not natively integrated with RAG retrieval pipelines, where the information retrieved (a ranked list of documents with metadata) is larger and more structured than scalar slot values. Memory-augmented RAG architectures, including MemoryBank [2.5.32] and LongMem [2.5.33], persist retrieved context across conversations using vector databases or compressed summaries, but they are designed for cross-session user modeling — remembering a user's preferences over weeks — rather than within-session retrieval deduplication. The entity resolution problem specific to multi-turn retrieval — determining whether a new utterance refers to a previously retrieved item and answering from memory rather than re-querying — has not been addressed by any prior RAG extension.

— Gap. Each extension surveyed in this subsection — query rewriting, post-retrieval evaluation, and multi-turn context persistence — has been evaluated independently as a point improvement on the standard RAG pipeline, on English benchmarks. The extensions share a structural absence: no prior work connects them into a control loop where the LLM functions as an active retrieval quality controller rather than a passive downstream consumer.

In such a loop, the LLM would inspect the query and decide on a retrieval strategy — a direct keyword lookup, a semantic similarity search, or a rewritten query — based on the query's lexical relationship to the document corpus. Retrieval would execute with quality gating: when all retrieval strategies produce noise, the pipeline would reject cleanly with empty results rather than feeding irrelevant documents to the LLM. The LLM would inspect the retrieved results, evaluate relevance against the original information need, and rephrase relevant items while detecting and gracefully reporting empty results rather than hallucinating from absent context. Multi-turn context would persist across turns, enabling the LLM to determine that a new utterance refers to a previously retrieved item and answer from memory rather than re-executing retrieval. Each of these capabilities has been demonstrated individually; none has been composed into an architecture where the LLM controls the retrieval process — deciding, evaluating, and adjusting — rather than merely consuming whatever retrieval returns.

This architectural gap is the position where the three disconnected extensions converge. Query rewriting operates pre-retrieval, post-retrieval evaluation operates post-retrieval, and multi-turn context operates across turns — but none provides a control signal that connects retrieval output to retrieval input. The standard pipeline has no position for a quality controller, and the extensions add functionality at the edges of the pipeline without closing the loop. The absence of a closed-loop retrieval architecture where the LLM actively controls retrieval quality — rather than passively receiving retrieved documents — is the gap addressed by the proposed method in §4.6.

— Overall Gap for §2.5. The RAG literature provides individually mature components at each pipeline stage — embedding models, indexing methods, and fusion techniques — surveyed in §2.5.2 through §2.5.4 as off-the-shelf selections with documented properties. The single architectural gap identified in this survey is the absence of a retrieval architecture where the LLM functions as an active quality controller in a closed loop, rather than a passive consumer of retrieved documents.

Prior extensions to the standard pipeline — query rewriting, post-retrieval evaluation, and multi-turn context — exist as disconnected point solutions, each evaluated in isolation on English benchmarks and each addressing one failure mode of the retrieve-then-generate paradigm. No prior work composes them into a unified architecture where the LLM decides on a retrieval strategy, retrieval executes with quality gating (rejecting cleanly when all strategies fail), the LLM inspects the retrieved results and evaluates their relevance against the original information need, relevant results are rephrased conversationally and empty results are reported without hallucination, and multi-turn context persists to prevent redundant retrieval cycles. The absence of this closed-loop control architecture — where the LLM acts as the controller in a feedback loop rather than a passive downstream element — is the research gap that motivates the knowledge retrieval pipeline proposed in §4.6.
