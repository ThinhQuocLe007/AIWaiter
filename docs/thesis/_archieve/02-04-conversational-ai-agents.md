Restaurant ordering is a conversational task with transactional consequences. Unlike a general-purpose question-answering session — where the output is information — a restaurant dialogue must produce state changes: items added to a cart, orders created in a database, payments processed, and robots dispatched. This section surveys the landscape in three stages. The first (§2.4.1) surveys four generations of technology that have been applied to restaurant dialogue — traditional NLU pipelines, LLM-based chatbots, commercial voice ordering systems, and physical delivery robots — describing what each achieves and where each falls short for Vietnamese restaurant deployment. The second (§2.4.2) surveys the foundational architectures for controlling LLMs in an agent loop — chain-based, autonomous reasoning-loop, and graph-based — and the tool-calling mechanism shared across them. The third (§2.4.3) surveys two challenges that any restaurant agent must solve regardless of architecture: classifying intent in informal Vietnamese speech, and preventing hallucinated LLM tool calls from reaching external systems.

### 2.4.1 LLMs, Chatbots, and Prior Restaurant Dialogue Systems

The problem of automating customer interaction in a restaurant — taking orders, answering questions about the menu, and processing payment — has been approached through several generations of technology, from rule-based slot-filling systems to large language model (LLM)-powered conversational agents. This section surveys the major approaches, describing what each achieves and where each falls short of the requirements of a Vietnamese restaurant AI waiter.

**Traditional NLU pipelines for task-oriented dialogue.** The earliest approach to automated restaurant ordering is the task-oriented dialogue system: a pipeline of natural language understanding (NLU), dialogue state tracking (DST), dialogue policy, and natural language generation (NLG) [2.4.1]. The NLU component performs two sub-tasks — intent classification (determining whether the user wants to order, search, or pay) and slot filling (extracting structured values like dish name, quantity, and modifiers from the utterance). For example, the utterance *"Cho 2 phần Ốc Hương Xốt Trứng Muối, ít cay"* yields the intent ORDER with slots {dish: "Ốc Hương Xốt Trứng Muối", quantity: 2, modifier: "ít cay"}.

NLU in these systems is typically implemented via supervised classifiers (SVM, CRF, or LSTM-based sequence taggers) trained on annotated dialogue corpora such as the DSTC (Dialog State Tracking Challenge) benchmarks [2.4.2]. The DST component maintains a belief state over slot values across turns, tracking which slots have been filled and which remain unresolved. The dialogue policy decides the next system action — ask for a missing slot, confirm the order, or proceed to payment. The NLG component produces a natural-language response from a template, e.g., *"Dạ, 2 phần Ốc Hương Xốt Trứng Muối, tổng 340.000đ. Xác nhận ạ?"*

This architecture has been deployed successfully in commercial restaurant ordering contexts for English, Chinese, Korean, and Japanese [2.4.3]. Platforms such as Rasa and Dialogflow provide open-source and cloud-hosted implementations of the NLU–DST–policy–NLG pipeline, enabling developers to build task-oriented dialogue systems with annotated training data and rule-based slot-filling logic [2.4.4]. The pipeline model's strength is its structure: every component has a well-defined input and output, the dialogue manager enforces valid state transitions, and the slot-filling mechanism guarantees that all required information is collected before an order is placed.

The pipeline model has four limitations for the Vietnamese restaurant context. First, the NLU classifier is language-specific — intent labels and slot types must be retrained for Vietnamese, requiring a dataset of annotated Vietnamese restaurant utterances which does not exist as a public resource. Second, the slot-filling mechanism handles a fixed schema of pre-defined slots (dish name, quantity, size, modifier) — but Vietnamese restaurant speech is more varied, with customers asking comparative questions ("Món nào ngon hơn?" — Which one tastes better?), expressing preferences ("Em thích ăn cay" — I like spicy food), and using vague descriptions ("món gì ấm bụng?" — What's warm and filling?). These utterances do not map cleanly to fixed slots. Third, the rule-based or SVM-based intent classifier is brittle to the informal Vietnamese and teencode variations common in spoken restaurant interactions (§2.4.3). Fourth, the architecture produces a structured API call (e.g., `create_order(item, qty)`) but does not integrate with the downstream systems a restaurant requires — a kitchen display, a robot fleet dispatcher, a payment gateway, and a real-time table management system.

**LLM-based chatbots and restaurant assistants.** The emergence of large language models (GPT-4, Qwen2.5, Gemini) [2.4.5] introduced a fundamentally different approach to restaurant dialogue: the LLM-powered chatbot. Rather than classifying intent and filling slots through a pipeline, an LLM processes the entire customer utterance and conversation history directly, generating a natural-language response. Combined with retrieval-augmented generation (§2.5), the LLM can ground its responses in the restaurant's actual menu — answering questions about dish ingredients, prices, and availability with apparent expertise [2.4.6].

Vietnamese-language chatbots have been developed by VNG (Zalo AI) and VinAI [2.4.7]. These systems demonstrate conversational Vietnamese capability across open domains — they can engage in multi-turn dialogue, answer factual questions, and maintain context across turns. In a restaurant context, such a chatbot could answer *"Ốc Hương có mấy loại?"* (How many types of snails?) with a list of sauce variants from the menu, or respond to *"Món nào ngon nhất quán?"* (What's the best dish?) with a recommendation based on popularity data. The chatbot's strength is flexibility — it handles the open-ended, conversational queries that the pipeline model's fixed slot schema cannot represent.

The fundamental limitation of the chatbot approach is that **dialogue does not equal transaction**. A restaurant interaction is not merely a question-answering session — it is a business process with material outcomes. When a customer says *"Cho 1 phần Ốc Hương Xốt Trứng Muối"*, the response *"Dạ, món đó giá 170.000đ ạ"* is linguistically correct but operationally empty — no item was added to a cart, no order record was created in the database, no kitchen display was updated, no payment was queued. The chatbot operates in the domain of text generation; the restaurant operates in the domain of state change. Bridging this gap requires an architecture where the LLM can produce not text but structured function calls — tool invocations that add items to a cart, create orders, request payment, and dispatch robots [2.4.8]. This transition from chatbot to agent is the subject of §2.4.2.

**Voice-based ordering systems.** Commercial voice-ordering systems represent a third approach — integrating speech recognition directly into the ordering flow. Wendy's FreshAI (USA, 2023) uses a custom LLM pipeline to process spoken drive-through orders in English, extracting menu items from unstructured speech and pushing them to the point-of-sale system [2.4.9]. Domino's AI-based voice ordering system (USA, 2019) similarly processes telephone orders through speech recognition and NLU, handling a fixed menu with structured customization options (pizza size, toppings, crust type). Both systems demonstrate that LLM-based voice ordering is commercially viable — they process real customer orders, reduce wait times, and integrate with existing kitchen and POS infrastructure.

However, these systems are narrow in three ways. They are English-only — the STT models, NLU classifiers, and generation components are trained and evaluated exclusively on English speech, and porting to Vietnamese would require retraining every component for a tonal language with fundamentally different acoustic and linguistic properties (§2.3). They are cloud-dependent — all inference runs on proprietary cloud infrastructure (Google Cloud for Wendy's, a custom platform for Domino's), meaning the system cannot operate during internet outages on the restaurant floor. They are stateless per transaction — each order is an independent session; there is no multi-turn memory across visits, no persistent cart that the customer can modify across utterances, and no customer preference tracking. And they have no physical delivery integration — the food is handed to the customer by a human server at a window or delivered by a human driver; the AI handles only the ordering conversation, not the fulfillment.

**Physical delivery robots.** The commercial restaurant robots surveyed in §2.1 — Bear Servi, Pudu Bellabot, Keenon T-series, and Alibaba Robot.He — represent the fourth category. These robots solve the physical delivery problem: navigate autonomously from kitchen to table, carry food on multiple tray levels, and present dishes to the customer. Their interaction interface is a touchscreen for button-based menu browsing and ordering, or a pre-recorded voice greeting that plays upon arrival. They have no conversational AI — no speech recognition, no natural language understanding, no dialogue management. They deliver food that was ordered through a separate channel (a human waiter, a tablet, or an app); they do not participate in the ordering conversation.

**Gap across all four categories.** Each category addresses one dimension of a Vietnamese restaurant AI waiter:

No prior system checks more than two of the five dimensions. The traditional NLU pipeline executes tools against a backend but lacks Vietnamese support and requires expensive retraining for the restaurant domain. LLM chatbots handle Vietnamese conversationally but cannot execute tools or affect system state. Voice ordering systems execute tools but are English-only and cloud-dependent. Academic restaurant dialogue systems demonstrate the pipeline approach for Asian languages but have no Vietnamese implementation and no physical robot integration. Physical delivery robots handle fulfillment but have no conversational capability.

**→ Gap.** No system combines Vietnamese language understanding, tool execution against a live backend, deterministic validation of LLM outputs, self-hosted deployment, and physical robot delivery. This is the high-level gap. The following subsections decompose it into three specific technical challenges: how to structure an agent that reliably transitions from LLM-generated text to validated system action (§2.4.2), how to classify Vietnamese restaurant utterances under teencode, context-dependent ambiguity, multi-intent compounding, and domain vocabulary (§2.4.3), and how to prevent the LLM from fabricating menu items that do not exist when generating tool call arguments (§2.4.4).

### 2.4.2 Architectures for Controlling LLMs

§2.4.1 established that chatbots — even linguistically capable Vietnamese chatbots — cannot affect system state. The mechanism that bridges conversation to transaction is **tool calling** (function calling): the LLM produces a structured invocation — a tool name and typed arguments — and the application framework executes the tool against live systems [2.4.10]. Tool calling was introduced by OpenAI as part of its Chat Completions API and has since been adopted by Anthropic (tool use), open-source serving frameworks (Ollama, vLLM), and agent frameworks (LangChain, LangGraph) [2.4.11].

Tool calling provides the *mechanism* for action; the *architecture* determines how that mechanism is governed — how the LLM interacts with tools, how the system decides what happens between LLM calls, and whether the agent can guarantee termination. Three foundational architectures have been proposed for LLM-based agents [2.4.12]. This section surveys each architecture descriptively: what it is, how it works, and what it provides. No architecture is evaluated for restaurant suitability here — that decision belongs in §4.3.1.

**Chain-based architecture (LangChain LCEL, prompt chaining).** The chain is the simplest architecture: the developer declares a fixed linear sequence of operations — prompt A → LLM call → extract field → prompt B → LLM call → output [2.4.13]. Each step is hard-coded; the LLM has no agency over which operation runs next. If the program defines a path for an ORDER utterance (classify → extract dish name → call `add_cart` → generate response), every ORDER utterance follows that same path. Strengths: the execution path is fully deterministic, predictable, and easy to debug. Each step produces a known output that feeds the next step. Limitation: the chain cannot branch. A SEARCH utterance that incorrectly arrives at the ORDER chain will proceed through ORDER steps — calling `add_cart` when the customer was asking about ingredients. There is no routing mechanism to redirect, no validation step to catch the mismatch, and no recovery path. The chain executes whatever input arrives at whatever step it was written to handle.

**Autonomous reasoning-loop architecture (ReAct, AutoGPT, AgentGPT).** The ReAct pattern interleaves reasoning and action in a loop [2.4.14]: the LLM observes the current state (conversation history + available tools), reasons about what to do, outputs either a tool call or a final answer, the framework executes the tool (if called) and appends the result to the observation, and the loop repeats until the LLM outputs a final answer. Unlike the chain, the LLM controls the sequence — it decides when to call tools, which tools to call, and when to stop. AutoGPT operationalized this pattern with internet access, file I/O, and long-term memory [2.4.18], spawning a class of autonomous agents where the LLM is the sole decision-maker at every step. Strengths: the architecture handles open-ended tasks where the number of reasoning steps is unknown in advance. It adapts to unexpected inputs because the LLM is always in the loop making decisions. Limitations: (a) **no guaranteed termination** — the LLM may loop indefinitely, calling tools repeatedly without converging on a final answer, (b) **no enforced validation between steps** — the LLM can call a tool with invalid arguments, observe garbage or an error, and continue reasoning as if nothing went wrong, and (c) **no explicit state machine** — there is no mechanism to enforce that step B must follow step A, or that step C is only legal after step B has completed successfully. The agent's behavior emerges from LLM generation, not from structural constraints.

**Multi-agent frameworks (AutoGen, CrewAI).** An extension of the autonomous loop, multi-agent frameworks distribute control across multiple LLM agents that communicate, delegate tasks, and coordinate autonomously [2.4.19]. One agent might handle information-gathering, another handles code execution, a third reviews output. The group reaches decisions through inter-agent conversation — consensus emerges from negotiation. Strengths: well-suited to creative, collaborative tasks (code generation with review, research with fact-checking, multi-perspective content creation). Limitations: the same lack of guaranteed termination and enforced validation that affects single-loop agents is multiplied across agents. A business process that must follow a fixed sequence (cart built → confirmed → paid) is inherently at odds with an architecture where the sequence emerges from agent negotiation — no individual agent is responsible for enforcing the process, and the group can collectively skip steps without any member flagging the omission [2.4.16].

**Graph-based architecture (LangGraph).** The graph-based architecture declares a directed state graph: a typed, shared state object flows through nodes connected by conditional edges [2.4.17]. Nodes can be LLM calls, tool executions, or deterministic functions. Edges are conditional — the next node depends on the output of the current one. Unlike the chain, edges can route to different nodes based on runtime conditions. Unlike the autonomous loop, the graph has a defined entry and exit — termination is structural, not emergent. Key capabilities that the graph structure enables: a **checkpointer** persists the shared state across graph invocations, providing multi-turn memory with per-thread isolation; **conditional edges** encode business logic as explicit routing decisions (if validation passes → proceed to tool execution, if validation fails → route back with corrective feedback); and **bounded retry with a circuit breaker** — a loop formed by conditional edges can iterate up to a hard-coded maximum and then exit, guaranteeing bounded execution regardless of LLM behavior. Because the graph topology is declared before execution, every possible path is known and bounded [2.4.20].

**→ Gap.** The three architectures provide progressively more control over LLM behavior — from zero (chain — no branching) to emergent (autonomous loop — LLM decides everything) to declared (graph — developer defines the state machine). None inherently enforces the combination that a restaurant agent requires: (a) guaranteed bounded execution — every turn must terminate with a defined outcome within a fixed maximum number of steps, (b) deterministic validation between every LLM call and external tool execution — no tool call may reach an external system without a prior deterministic check against known-good data, and (c) explicit business-process enforcement — the system must guarantee that the ordering sequence (cart built → confirmed → paid) is followed, and that invalid transitions (confirming an empty cart, paying before ordering) are structurally impossible. Tool calling provides the mechanism; a chosen architecture must provide the governance. This gap — the need for an architecture that structurally enforces these three properties — motivates the graph-based agent design in §4.3.1, where the graph topology directly encodes the restaurant ordering state machine.

### 2.4.3 Challenges Any Restaurant Agent Must Solve

> *A restaurant agent must classify the customer's intent in informal Vietnamese and must prevent hallucinated tool calls from reaching external systems. These challenges are orthogonal to the architecture choice in §2.4.2 — they must be solved regardless of whether the agent is chain-based, loop-based, or graph-based. Each challenge is surveyed below: what it is, how existing approaches (fail to) handle it, and what gap remains.*

#### (a) Intent Classification in Informal Vietnamese Restaurant Speech

§2.4.2 described the architecture that governs tool execution. But before the agent can execute any tool, it must determine *which* tool the customer's utterance demands. A customer who says *"Ốc Hương có mấy loại?"* (How many types of snails do you have?) is searching; *"Cho 2 Ốc Hương đi"* (Give me two snails) is ordering; *"Tính tiền"* (Check please) wants to pay; *"Cảm ơn em"* (Thank you) is making conversation. Routing an ORDER utterance to the search subsystem produces irrelevant menu results instead of a cart update; routing a PAYMENT utterance to the chat subsystem produces polite conversation instead of a payment bill [2.4.22].

Intent classification in a restaurant agent is a 4-class decision problem: given a Vietnamese utterance and the current conversation state, select the correct intent from {ORDER, SEARCH, PAYMENT, CHAT}. This appears to be a standard text classification problem, solvable by the many mature classification approaches available. However, Vietnamese restaurant speech introduces four domain-specific challenges that each prior approach fails to address simultaneously.

**Four domain-specific challenges.**

**(i) Teencode and informal Vietnamese.** Casual Vietnamese restaurant speech differs substantially from the formal written Vietnamese that dominates NLP training corpora [2.4.23]. Common informal variants in restaurant ordering include: *"ad"* for *anh/chị* (sir/madam), *"ck"* for *chuyển khoản* (bank transfer), *"z"* for *vậy* (so/then), *"nhiêu"* for *bao nhiêu* (how much), *"hông"* for *không* (no/not), and *"vs"* for *với* (with). A customer saying *"ck cho mình cái QR với"* (transfer me the QR code) uses three informal variants in a 6-word utterance. Traditional intent classifiers trained on formal Vietnamese corpora have never seen these tokens — the classifier cannot recognize *"ck"* as a payment-intent keyword because *"ck"* does not appear in its training vocabulary. Even when the classifier is retrained, new informal variants emerge organically in spoken language — the classifier must adapt without requiring a new training run for every new slang term.

**(ii) Context-dependent ambiguity.** Short affirmations carry different intents depending on the conversation stage [2.4.24]. When the agent asks *"Dạ, anh/chị xác nhận đặt món chưa ạ?"* (Would you like to confirm the order?), the customer's reply of *"ok em"*, *"ừ"*, or *"đúng rồi"* must route to ORDER_CONFIRM. The identical utterance at the start of a conversation — the customer has just been seated and says *"ok em"* in response to a greeting — must route to CHAT. A classifier that sees only the utterance text, without access to the conversation state (current order stage, cart contents, prior dialogue turns), cannot distinguish these cases. The text *"ok em"* carries zero intent signal by itself — all the signal is in the context.

**(iii) Multi-intent compounding.** Vietnamese speakers frequently express multiple actions within a single utterance [2.4.25]. *"Cho 2 Ốc Hương rồi tính tiền luôn"* (Give me two snails and then the bill) combines ORDER and PAYMENT. *"Lẩu Thái cay không? Không cay thì cho 1 phần"* (Is the Thai hot pot spicy? If not, give me one) combines SEARCH and ORDER. Single-label classifiers — the vast majority of production text classifiers — force a single output label, collapsing a two-intent utterance into whichever intent the classifier deems dominant. The customer expected both actions; the system executed only one. Multi-label classifiers exist but require per-intent sub-query extraction — *"Cho 2 Ốc Hương rồi tính tiền luôn"* must be decomposed into the ORDER fragment (*"Cho 2 Ốc Hương"*) and the PAYMENT fragment (*"tính tiền luôn"*) before each can be processed by its respective subsystem. Classification alone, without decomposition, is insufficient.

**(iv) Domain vocabulary.** Restaurant dish names are complex noun phrases out-of-distribution for general-domain NLP models [2.4.26]. Names like *"Ốc Hương Xốt Trứng Muối"* (Spicy Salted Egg Snail), *"Lẩu Cá Tầm Măng Chua"* (Sturgeon Hot Pot with Sour Bamboo Shoots), and *"Sò Điệp Nướng Mỡ Hành"* (Grilled Scallops with Scallion Oil) combine food nouns, preparation methods, and sauce types into multi-word compounds that appear only in restaurant contexts. An embedding model trained on general-domain Vietnamese text (news, Wikipedia, social media) produces embeddings for these dish names that are closer to unrelated centroid regions than to the correct intent centroid. A customer saying *"Cho 1 Lẩu Cá Tầm Măng Chua"* may produce an ORDER-intent embedding weaker than a general-domain phrase like *"Tôi muốn đặt món"* (I want to order) — the dish name confuses the embedding space rather than reinforcing it.

**Traditional NLU pipelines (Rasa, SVM-based classifiers).** Rule-based and classical ML intent classifiers rely on keyword matching, regular expressions, and n-gram features extracted from formal Vietnamese corpora [2.4.27]. They are fast (~5ms per classification) and have been deployed at scale. Their failure on Vietnamese restaurant speech is threefold: (a) keyword lists must be manually curated and cannot anticipate informal variants — *"ck"* is not in any formal Vietnamese payment-intent keyword list, (b) SVM decision boundaries are trained on text features alone, with no mechanism to incorporate conversation state, and (c) each new dish name added to the menu requires retraining or rule updates.

**Lightweight text classifiers (fastText, SetFit).** FastText uses subword n-gram features to classify text, giving it some robustness to unseen words (a property that helps with rare dish names) [2.4.28]. SetFit fine-tunes a sentence transformer with a small number of labeled examples per class, achieving competitive accuracy with few-shot data [2.4.29]. Both are fast (~2–10ms) and trainable on domain-specific data. However, both share the same fundamental limitation: they classify text in isolation. Without access to conversation state, a fastText model cannot distinguish *"ok em"* at confirmation from *"ok em"* at greeting — both produce the same n-gram feature vector. FastText's subword features provide partial coverage for unseen teencode variants (character-level n-grams may capture *"ck"* as a substring) but provide no protection against systematic informal-to-formal mapping failures. And while both can be fine-tuned on a restaurant-specific dataset, the effort of collecting, labeling, and retraining for every menu update and every emergent teencode pattern is an operational burden that a production restaurant system cannot sustain.

**Semantic centroid routing.** A popular lightweight alternative is centroid-based semantic routing: compute a sentence embedding for the utterance, compare it against pre-computed per-intent centroid vectors via cosine similarity, and select the intent with the highest similarity score [2.4.30]. This approach is fast (~15ms after embedding), requires no training (only centroid computation from labeled examples), and adapts to new vocabulary automatically — new dish names are encoded by the same embedding model, so they participate in similarity computation without retraining. It handles challenge (iv) reasonably well.

However, centroid routing fails fundamentally on challenges (i) through (iii). Teencode variants produce embeddings that are distant from their formal equivalents — *"ck cho mình cái QR"* is far from the PAYMENT centroid in embedding space because the embedding model was not trained on informal Vietnamese. Context-dependent utterances cannot be disambiguated because the embedding of *"ok em"* is identical regardless of conversation stage — and the centroids for ORDER and CHAT are both approximately equidistant from *"ok em"* in embedding space. Multi-intent utterances produce a single embedding that blends two different semantic signals, potentially matching neither centroid well enough to pass a confidence threshold. The centroid router's published accuracy on domain-specific classification benchmarks typically ranges from 75–90%, with the gap concentrated precisely in these three failure modes [2.4.31].

**LLM-based routing (few-shot).** Routing via an LLM — prompt the model with the utterance and ask it to classify — is the most capable approach on paper. An LLM prompted with few-shot examples can handle teencode (it has seen informal Vietnamese in its pretraining data), can process conversation context (last N turns injected into the prompt), can output multi-intent lists (by instructing the model to return a JSON array), and can understand domain vocabulary (dish names in the prompt are tokenized and attended to normally) [2.4.32].

The problem is latency and determinism. An LLM call for classification — even with a small model (3B parameters) and a short prompt — takes approximately 1.5–2.0 seconds on consumer-grade GPU hardware. This means the customer waits 2 seconds before the agent even begins deciding what action to take. Classification latency adds to agent inference latency, tool execution latency, and response generation latency — a 2-second classifier turns a 4-second voice turn into a 6-second turn. For the 60–70% of utterances that are straightforward single-intent cases, this latency is wasted — the utterance is trivially classifiable and does not need the LLM's reasoning capacity.

The determinism problem is subtler. The same utterance classified twice by an LLM may produce different results because of temperature sampling and the non-determinism of GPU floating-point arithmetic [2.4.33]. For a restaurant, determinism matters: *"ok em"* must always route to ORDER when the customer is confirming, without any probability of routing to CHAT. An LLM-based router with temperature 0.0 reduces but does not eliminate this non-determinism because the underlying GPU operations are still non-deterministic in most inference frameworks.

**→ Gap.** Each prior classification approach fails on at least two of the four challenges, and no approach simultaneously satisfies all four while maintaining sub-millisecond inference latency. LLM routing satisfies the accuracy requirements but fails on latency and determinism. Semantic centroid routing satisfies latency but fails on teencode and context. Traditional and lightweight classifiers satisfy latency but fail on teencode, context, and domain vocabulary. The gap is for a classification approach that combines (a) the domain knowledge and teencode handling of an embedding model trained on Vietnamese, (b) the context awareness of conversation state features (order stage, cart contents, search history), (c) the accuracy of a trained decision boundary that learns which context features matter for which ambiguous cases, and (d) the determinism and sub-millisecond latency of a forward pass through a small neural network — with no LLM call required for the common case.

#### (b) Post-Generation Validation — Preventing Hallucinated Tool Calls

Even when intent is classified correctly and the correct worker node is activated, the LLM that generates the tool call arguments produces text probabilistically — and probabilistic generation includes hallucination [2.4.34]. The LLM may fabricate a dish name not on the menu — a customer orders Vietnamese seafood and the LLM produces `add_cart(name="Pizza Hải Sản", quantity=1)`, a nonsensical combination that exists nowhere. It may produce an absurd quantity — `add_cart(name="Ốc Hương", quantity=999)`. It may attempt to confirm an order containing items that failed validation on the previous turn, because the LLM's per-call statelessness means it has no memory of the validator's prior rejection. Hallucination is not a bug in any specific LLM — it is an inherent property of autoregressive text generation. The model samples from a probability distribution over its vocabulary; improbable tokens are sampled occasionally, and when they appear in a structured output (tool call arguments), they become wrong actions.

**The restaurant stakes.** The consequences of hallucination differ fundamentally between a chatbot and a transactional agent [2.4.35]. In a chatbot, a hallucinated response is an annoyance — the customer receives wrong information, realizes the error, and corrects it. The interaction cost is a repeated utterance. In a restaurant agent, a hallucinated tool call has material consequences: a wrong dish name in `add_cart` becomes a wrong item in the customer's bill. A hallucinated `confirm_order` with invalid items sends an order to the kitchen for food the customer did not request — the kitchen cooks it, resources are wasted, and the customer is charged incorrectly. A hallucinated `request_payment` for the wrong amount overcharges or undercharges a real customer. The cost of hallucination is measured in money, food waste, and customer trust.

**Existing mitigation approaches and their limits.** Three strategies exist for reducing hallucination risk, each with a fundamental limitation:

**(i) Constrained decoding / structured output.** The LLM is forced to output valid JSON matching a predefined schema — field names, types, and required fields are enforced at the token generation level. This prevents malformed tool calls (`add_cart(name=, quantity="many")`) but does not validate semantic correctness. `{"tool": "add_cart", "arguments": {"name": "Pizza Hải Sản", "quantity": 1}}` is valid JSON with a string name and an integer quantity — the schema is satisfied, the semantics are wrong. Schema enforcement catches syntax errors, not factual errors.

**(ii) Retrieval-augmented generation (RAG).** Grounding the LLM in the actual menu documents reduces the probability of hallucination by giving the model access to the correct dish names at generation time. If the menu contains *"Ốc Hương Xốt Trứng Muối — 170.000đ"* and this text is in the LLM's context window, the model is more likely to generate that dish name than to fabricate one. However, RAG reduces probability; it does not eliminate it. The LLM may still ignore or misread the retrieved context — a known failure mode where the model attends to the wrong part of the prompt or defaults to its pretraining knowledge over the retrieved document. And RAG is a generation-time strategy: the retrieved documents influence what the model produces but do not validate what it actually produced

**(iii) Human-in-the-loop approval.** A human reviews each tool call before execution. This eliminates hallucination risk entirely — if a human must approve every order, no wrong order reaches the kitchen. However, this is incompatible with autonomous restaurant operation. The purpose of an AI waiter is to handle ordering without requiring a human server for every interaction. A human babysitting every LLM tool call provides safety at the cost of the very autonomy the system was designed to provide.

**Generation-time vs. post-generation validation.** All three mitigation strategies operate at generation time — they constrain what the LLM can output, either by limiting the token space (structured output), influencing the probability distribution (RAG), or requiring external approval (human-in-the-loop). None validates the LLM's output *after* generation against an external, authoritative source of truth. This distinction — constrain what the LLM can say vs. check what the LLM actually did say — is the critical architectural gap.

A post-generation validator would receive the LLM's tool call as input and check every argument against known-good data *before* the tool executes: does this dish name exist in the menu database? Is this quantity within a reasonable range (1–20, not 999)? Is this state transition valid (confirming requires the cart to be non-empty and the stage to be AWAITING_CONFIRMATION)? If any check fails, the tool call is blocked, corrective feedback is sent to the LLM for a retry, and a circuit breaker prevents infinite retry loops. The LLM is allowed to make mistakes — the validator catches them before they become wrong orders.

**→ Gap.** No prior restaurant agent implements a deterministic post-generation validation layer that checks every LLM tool call argument against an external source of truth (the actual menu database, the actual cart state, the valid state transitions) before the call reaches external systems. All existing approaches attempt to prevent hallucination at generation time through constraints, retrieval, or approval; none validates the generated output after the fact. This gap is the motivation for the deterministic validator designed in §4.3.4.

---

### → Gap (Overall for §2.4)

No prior system simultaneously provides: (a) Vietnamese language support — understanding and generating Vietnamese across the full restaurant ordering lifecycle, (b) an agent architecture that structurally enforces deterministic validation between every LLM call and tool execution, guaranteed bounded execution with circuit-breaker termination, and explicit business process state-machine enforcement — properties that no existing architecture (chain, autonomous loop, multi-agent, or graph) provides natively for a restaurant context, (c) intent classification that handles all four Vietnamese-specific challenges — teencode, context-dependent ambiguity, multi-intent compounding, and domain vocabulary — with sub-millisecond latency and deterministic output, (d) a deterministic post-generation validation layer that catches LLM hallucination in tool call arguments — fabricated dish names, invalid quantities, impossible state transitions — before those calls reach external systems, and (e) self-hosted deployment — all components running on on-premises hardware with no cloud API dependency.

The agent architecture proposed in §4.3 is organized around these five gaps. The LangGraph StateGraph (§4.3.1) satisfies (b) by encoding the restaurant ordering state machine directly in the graph topology — conditional edges define legal transitions, a checkpointer persists state across turns with session isolation, and a circuit breaker guarantees bounded execution. The trained MLP intent classifier (§4.3.2) satisfies (c) by combining a frozen Vietnamese bi-encoder embedding with 10 hand-crafted context features (order stage, cart state, search history) — a 778-dimensional input to a 3-layer MLP that classifies in ~0.17ms. The deterministic validator (§4.3.4) satisfies (d) by acting as a mandatory node between every LLM call and every tool execution — it resolves dish names against the menu database, detects off-menu items, flags ambiguous names, and blocks tool calls that would violate the state machine. Vietnamese support (a) is provided by the entire pipeline operating in Vietnamese — from PhoWhisper STT and Vietnamese bi-encoder embeddings through to Vietnamese system prompts and template responses. Self-hosted deployment (e) is achieved through Ollama serving Qwen2.5 7B on an on-premises GPU server with no external API calls.
